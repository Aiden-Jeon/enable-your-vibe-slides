    학습 목표

    • Claude Code CLI의 설치와 기본 사용법을 숙지한다
    • CLAUDE.md를 활용한 프로젝트 컨텍스트 관리법을 이해한다
    • 주요 명령어와 실용 팁을 학습한다

    학습 목표를 하나씩 짚어봅니다. 이 섹션이 끝나면 여러분은 Claude Code를 직접 설치하고, 터미널에서 자연어로 코드를 작성할 수 있게 됩니다. CLAUDE.md 파일을 통한 컨텍스트 관리도 핵심 내용입니다.

    Claude Code란?

    CLI 기반 AI 코딩 어시스턴트

    • 터미널에서 직접 실행하는 에이전틱 코딩 도구
    • 자연어로 코드 생성, 수정, 디버깅
    • 파일 시스템, 터미널, 웹에 직접 접근
    • 기존 개발 환경을 바꾸지 않고 사용

    설치 방법

    1단계: Node.js 확인

    # Node.js 18+ 필요
node --version
# v20.x.x 이상 권장

    2단계: Claude Code 설치

    # npm으로 전역 설치
npm install -g @anthropic-ai/claude-code

    3단계: 인증

    # 최초 실행 시 브라우저에서 Anthropic 계정 인증
claude
    Tip: 설치 후 claude --version으로 정상 설치 확인
    우선 설치가 되어 있는 거는 저희가 최근에 나왔었던 Vibe라고 하는 프로젝트를 통해서 설치하실 수도 있고요. 그거 설치하게 되면 기본적으로 Claude가 같이 설치가 됩니다. 혹시 아직 설치 안 돼 있으면 Node.js 사용하셔가지고 설치하시고 claude 실행되는지 한번 봐주시면 될 것 같습니다.

    Endpoint 설정 (1/3)

    Step 1. Databricks Workspace에서 Coding Agent Integration 설정

    Claude Code를 선택하고 사용할 모델(Sonnet, Opus 등)을 지정합니다.
    그러면 이제 endpoint에 대한 설정이 필요한데, 저희 회사에서 제공하는 것들을 사용할 수 있습니다. Vibe Coding에 대해 제공을 한다고 하고 회사에서 홍보를 하고 있는데, 실제로 한번 사용해 보겠습니다. 서빙 쪽에 들어오셔서 Other Integration 쪽에 보시면 Claude Code를 선택하실 수 있습니다.

    Endpoint 설정 (2/3)

    Step 2. settings.json 설정

    Generate API Key 버튼으로 토큰을 생성하고 설정에 반영합니다.
    여기에서 Generate API Key를 통해서 만들어주시면 됐고요.

    Endpoint 설정 (3/3)

    Step 3. 프로젝트 레벨 설정

    프로젝트 루트의 .claude/ 디렉토리에 settings.local.json 설정 파일을 생성합니다.

    클립보드에서 바로 설정하기

    # 설정 JSON을 클립보드에 복사한 후 실행
bash setup-endpoint.sh
    Tip: 스크립트가 클립보드의 JSON 유효성을 검증하고 .claude/settings.local.json 파일을 자동으로 생성합니다. 기존 파일이 있으면 백업도 해줍니다.
    Claude Code 같은 경우에는 글로벌에서 관리가 되는 변수가 있고 그리고 각 프로젝트 단위로 되는 설정들이 있습니다. 저희가 이 설정 자체를 글로벌하게 할 거는 아니기 때문에 프로젝트 단위로 설정을 할 거고요. 프로젝트에서는 프로젝트 루트의 settings.local.json을 설정하면 됩니다. 아까 클립보드 설정하시고 처음에 설정했었던 레포를 기준으로 들어가시면 setup-endpoint라는 게 있을 겁니다. 그러면 자동으로 설정을 해줍니다.

    Endpoint 설정 확인

    claude를 실행하여 Databricks endpoint가 정상 연결되었는지 확인합니다.

    claude
    ╭─── Claude Code ──────────────────────────────╮

│                  ▐▛███▜▌                     │

╰──────────────────────────────────────────────╯
    하단에 databricks-claude-opus-4-6이 표시되면 Databricks endpoint 연결 성공!
    claude 명령을 쳤을 때 보시는 것처럼 databricks-claude-opus라고 해서 databricks라는 prefix가 잘 붙어 있으면 잘 설정이 된 걸 볼 수 있습니다. 여기서 그냥 "하이" 이런 식으로 간단하게 물어봐 주시면 될 것 같습니다. 설정이 되는지 확인해 봅니다.

    그다음에 터미널에 대한 명령어를 쓸 수 있습니다. 터미널을 쓰는 방법 같은 경우에 크게 두 가지가 있습니다. 예를 들어서 "실행해줘"라고 하는 식으로 실행해 줄 수 있고, 혹은 명령어를 던질 수 있는데 만약에 Claude Code가 내가 원하는 것들을 좀 찾기 힘들고 내가 그 컨텍스트 다 설명하기 싫다라고 하는 경우에는 Bash Mode라고 하는 게 있습니다. 느낌표(!)를 주시면 Bash Mode가 실행이 되고요, 여기서는 쉽게 사용하는 명령어들 사용하셔도 됩니다. 내가 뭔가 보고 싶어서 나가고 싶다라고 하는 경우에는 Ctrl+C 버튼을 누르게 되면 bash 셸로 나갔다가 fg 누르게 되면 다시 세션을 실행시킬 수 있는 옵션도 있습니다. 이렇게 실행시키게 되면 실행 결과나 이런 것들이 컨텍스트에 남기 때문에 이 컨텍스트를 기준으로 다시 Claude한테 어떤 작업을 해달라고 요청을 하실 수도 있습니다.

    CLAUDE.md

    프로젝트 컨텍스트 파일

    CLAUDE.md란?

    • 프로젝트 루트에 위치하는 컨텍스트 파일
    • Claude Code가 세션 시작 시 자동으로 읽음
    • 프로젝트의 규칙, 구조, 스타일 가이드 정의
    • 팀원 간 AI 사용 일관성 보장
    • /init 명령으로 자동 생성 가능
    # CLAUDE.md

## 프로젝트 개요
Python 3.11 기반 FastAPI 백엔드

## 코딩 규칙
- Black 포매터 사용 (line-length=88)
- 모든 함수에 타입 힌트 필수
- docstring은 Google 스타일

## 테스트
- pytest 사용
- tests/ 디렉토리에 위치
- 커버리지 80% 이상 유지

## 주의사항
- .env 파일은 절대 커밋하지 않음
- API 키는 환경변수로 관리
    그리고 CLAUDE.md라고 하는 파일이 있습니다. CLAUDE.md 같은 경우에는 이 프로젝트에서 Claude가 어떤 컨텍스트를 참조하는 순서들이 있는데, 예를 들어서 CLAUDE.md가 없는 경우에는 전체 파일을 한번 스캔해서 이 명령을 수행하기 위한 것들이 어디에 있는지 탐색을 해야 됩니다. 그렇게 되면 컨텍스트를 조금 많이 사용을 하게 되는데, CLAUDE.md에 넣어 놓으면 Claude가 "아 이쪽에 이런 게 있겠구나"라고 해서 탐색하는 범위를 줄일 수 있습니다. 그래서 Claude로 어떤 협업을 하는 케이스에서는 CLAUDE.md 파일을 잘 만들어 놓게 되면 다른 사용자들, 개발자들 간에서 협업을 할 때 Claude를 사용한다 하더라도 같은 방법으로 사용할 수 있게 됩니다. 특히 /init 명령어를 사용하게 되면 자동으로 생성을 할 수가 있습니다.

    컨텍스트 관리

    파일 참조 방법

    # 파일 경로를 직접 언급
> "src/utils/helpers.py를 보고 비슷한 패턴으로 새 유틸 함수 만들어줘"

    @ (파일 경로 자동완성)

    # @를 입력하면 파일 경로 자동완성 활성화
> @src/utils/helpers.py 이 파일을 참고해서 새 유틸 만들어줘
> @tests/ 기존 테스트 패턴을 따라서 새 테스트 작성해줘
    @를 입력하면 파일 경로 자동완성이 활성화됩니다. 특정 파일을 빠르게 참조할 수 있습니다.
    그리고 내가 예를 들어서 "찾아줘"라는 것보다는 내가 특정하게 딱 이걸 하고 싶다라고 하면 @(골뱅이)로 참조할 수가 있고, @해서 보시면 이렇게 컨텍스트를 참조를 해서 "뭘 해줘" 이런 식으로 빠르게 참조해서 쓸 수 있습니다.

    컨텍스트 우선순위

    1 CLAUDE.md — 자동 로드되는 프로젝트 설정
    2 현재 대화의 이전 메시지
    3 명시적으로 참조한 파일 (@로 지정)
    4 Claude가 자동 탐색한 파일
    컨텍스트의 우선 순위는 아까 말씀드린 것처럼 CLAUDE.md 파일이 제일 처음 읽고요. 그리고 이 세션 내에서 있었던 이전 메시지, 그리고 저희가 참조한 거, 그다음에 마지막으로 여기에서 충분하지 않은 경우에 Claude가 도구들을 사용해서 자동으로 탐색한 파일들을 보게 됩니다.

    품질 저하 구간
    컨텍스트 윈도우의 앞쪽 30%에서 최고 성능 → 채워질수록 응답 품질이 떨어집니다
    컨텍스트 윈도우에 대해서 이야기를 해보겠습니다. LLM은 컨텍스트 윈도우라는 제한된 크기의 메모리를 가지고 있는데요, 이 윈도우가 채워질수록 성능이 저하됩니다. 처음 30% 정도에서는 최고 성능을 보이고, 그 다음 30%에서는 양호하지만, 마지막 40% 구간에 들어가면 응답 품질이 눈에 띄게 떨어집니다. 이건 모델의 구조적인 한계이기 때문에, 컨텍스트를 어떻게 관리하느냐가 매우 중요합니다.

    그러면 내가 지금 컨텍스트를 얼마나 쓰고 있는지 어떻게 알 수 있느냐. /context 명령어를 사용하면 현재 사용량을 확인할 수 있습니다. 여기 보시면 전체 200k 토큰 중에서 67%를 사용하고 있다는 걸 볼 수 있고, 대화 내용이 가장 많은 비중을 차지하고 있죠. 일반적으로 60%를 넘어가기 시작하면 /compact로 압축하거나, /clear로 새 세션을 시작하는 걸 추천드립니다.

    자율 코딩 패턴

    컨텍스트 한계를 극복하는 반복 전략

    랄프 루프

    Ralph Loop — Geoffrey Huntley

    • 매 반복마다 컨텍스트를 리셋하고 파일/git에 진행 저장
    • 테스트·타입체크 등 피드백 루프로 품질 보장
    • Ralph Wiggum처럼 끈질기게 반복하여 결국 성공

    시지푸스

    Sisyphus — oh-my-opencode

    • 끊임없이 반복하는 끈기의 오케스트레이터
    • 전문 에이전트들에게 위임·병렬 실행
    • 계획과 실행을 분리하여 컨텍스트 효율 극대화

    공통 핵심: 컨텍스트 포화 전에 리셋·위임하여 항상 Smart Zone에서 작업

    Claude Code에서: /clear로 새 세션 시작 · /compact로 압축 · 작업을 작은 단위로 분할
    컨텍스트 윈도우의 한계를 극복하는 자율 코딩 패턴 두 가지를 소개하겠습니다. 첫 번째는 시지푸스입니다. oh-my-opencode라는 오픈소스 프로젝트에서 만든 프레임워크인데요, 그리스 신화의 시지푸스가 끊임없이 바위를 올리는 것처럼, 포기하지 않고 반복하는 끈기를 상징합니다. 시지푸스는 메인 오케스트레이터 역할을 하면서 전문 에이전트들에게 작업을 위임합니다. 예를 들어 코드 탐색은 Explorer에게, 아키텍처 설계는 Oracle에게, 구현은 Hephaestus에게 맡기는 식이죠. 계획과 실행을 분리하고 병렬로 실행해서 컨텍스트를 효율적으로 씁니다. 두 번째는 랄프 루프입니다. Geoffrey Huntley가 만든 패턴이고, 심슨 가족의 Ralph Wiggum에서 이름을 따왔습니다. 핵심 아이디어는 매 반복마다 컨텍스트를 완전히 리셋하는 겁니다. 진행 상황은 파일과 git 히스토리에 저장하고, 테스트와 타입체크로 품질을 보장합니다. oh-my-opencode에서도 Ralph Loop를 내장하고 있는데, 원래 구현은 하나의 세션에서 계속 돌려서 컨텍스트가 차는 문제가 있었는데, 지금은 매 반복마다 fresh context로 리셋해서 항상 Smart Zone, 즉 컨텍스트의 앞쪽 40%에서 작업하도록 개선됐습니다. 두 패턴의 공통점은 컨텍스트가 포화되기 전에 리셋하거나 위임한다는 겁니다. 우리가 Claude Code에서 /clear나 /compact를 쓰는 것도 같은 원리예요. 작업이 길어지면 컨텍스트를 정리하고, 큰 작업은 작은 단위로 나눠서 진행하는 게 핵심입니다.

    커스텀 VLIW SIMD 프로세서에서 커널 최적화하는 채용 시험

    Claude Opus 4.5가 대부분의 인간 지원자를 이기자 → 시험을 오픈소스로 공개

    "1,487 cycles 이하로 최적화하면 이력서를 보내달라" — Anthropic

    재미있는 사례를 하나 소개드리자면, Anthropic에서 성능 엔지니어를 뽑을 때 쓰던 채용 시험이 있었습니다. 커스텀 프로세서 시뮬레이터에서 커널 코드를 최적화하는 문제인데요, 기본 코드가 147,734 cycles이 걸리는 걸 얼마나 줄일 수 있느냐가 과제였습니다. 그런데 Claude Opus 4.5가 이걸 1,487 cycles까지, 약 99배 속도 향상을 달성해서 대부분의 인간 지원자를 이겨버렸습니다. 그래서 Anthropic은 이 시험을 GitHub에 오픈소스로 공개하면서 "Claude보다 더 잘 풀면 이력서를 보내달라, 채용을 논의하겠다"는 도전장을 냈죠. 이게 가능했던 건 AI가 충분한 컨텍스트 안에서 반복적으로 최적화할 수 있었기 때문입니다. 반대로 컨텍스트가 포화되면 이런 반복 최적화 자체가 불가능해지고, 앞서 본 시지푸스 코딩이나 랄프 코딩에 빠지게 되는 겁니다.

    랄프 루프의 실전 사례

    채용 시험에서 Claude가 실제로 돌린 반복 과정

    1. Read
    코드 & 아키텍처
    문서 분석
    →
    2. Profile
    시뮬레이터 실행
    병목 지점 식별
    →
    3. Optimize
    최적화 기법 적용
    코드 수정
    →
    4. Measure
    cycle 수 측정
    결과 비교
    ↻ 개선 여지가 있으면 2~4 반복
    SIMD 벡터화 — 데이터 병렬 처리로 여러 연산을 한 사이클에
    루프 언롤링 — 반복문 풀어서 분기 예측 오버헤드 제거
    명령어 스케줄링 — VLIW 파이프라인에 맞게 명령어 재배치
    메모리 접근 최적화 — 캐시 히트율 높이는 데이터 배치
    핵심: 앞에서 본 랄프 루프 그 자체 — "코드 읽기 → 실행 → 결과 분석 → 수정"을 반복하며 147,734 → 1,487 cycles로 최적화
    Claude가 이 시험을 어떻게 풀었는지 좀 더 상세하게 보겠습니다. Claude가 돌린 사이클은 크게 4단계입니다. 첫째, Read — 프로세서 아키텍처 문서와 기존 커널 코드를 읽어서 VLIW SIMD 프로세서의 제약 조건을 파악합니다. 이 프로세서가 몇 개의 실행 유닛을 가지고 있고, 어떤 명령어를 동시에 실행할 수 있는지, 파이프라인 구조가 어떤지를 이해하는 거죠. 둘째, Profile — 시뮬레이터를 실제로 돌려서 현재 몇 cycle이 걸리는지 측정하고, 어디서 병목이 발생하는지 식별합니다. 어떤 루프에서 가장 많은 cycle을 소모하고 있는지, 파이프라인 스톨이 어디서 발생하는지를 분석합니다. 셋째, Optimize — 식별한 병목에 맞는 최적화 기법을 적용합니다. SIMD 벡터화로 데이터를 병렬 처리하고, 루프 언롤링으로 분기 오버헤드를 줄이고, VLIW 아키텍처에 맞게 명령어를 재배치해서 파이프라인을 최대한 활용하고, 메모리 접근 패턴을 최적화해서 캐시 히트율을 높였습니다. 넷째, Measure — 수정한 코드를 다시 시뮬레이터에 돌려서 cycle 수가 얼마나 줄었는지 확인합니다. 이 2~4단계를 개선 여지가 있을 때까지 반복한 겁니다. 이게 바로 앞에서 배운 랄프 루프의 실전 사례입니다. 매 반복마다 시뮬레이터 실행 결과라는 피드백 루프가 있고, 진행 상황은 코드 파일에 저장되니까 컨텍스트를 리셋해도 이어갈 수 있는 거죠. Claude Code가 파일을 읽고, 터미널에서 시뮬레이터를 실행하고, 결과를 분석해서 코드를 수정하는 — 이 전체 사이클을 자동으로 돌린 겁니다. 결국 147,734 cycles에서 시작해서 1,487 cycles까지, 약 99배의 성능 향상을 달성했습니다.

    Section 03: MCP Architecture

    Claude Code의 기능을 확장하는 MCP(Model Context Protocol)의 구조와 원리를 배웁니다.

    이상입니다.

    Databricks와 상호작용하는 방법들

    방법 설명 Claude Code 활용
    Databricks Connect 로컬 IDE에서 원격 클러스터로 Spark 코드 실행 코드 생성 & 실행
    Databricks CLI 터미널에서 워크스페이스 리소스 관리 Skills 만들기에 최적!
    Databricks SDK Python/Java/Go SDK로 프로그래밍 방식 제어 코드 생성 & 실행
    REST API HTTP 요청으로 모든 기능에 직접 접근 MCP 서버 구축

    💡 최근 "MCP vs CLI" 논쟁이 있지만, Databricks는 CLI·SDK·API를 모두 갖추고 있어 상황에 맞게 선택 가능

    이 도구들을 활용해 MCP 서버와 Skills를 만들어보자!

    Claude Code에서 Databricks 작업을 하려면 여러 방법이 있습니다. Connect, CLI, SDK, REST API — Databricks는 이 모든 인터페이스를 다 제공하고 있죠. 최근에 MCP보다 CLI가 낫다는 논쟁이 있는데, 사실 Databricks는 다 갖추고 있어서 이 논쟁과 크게 상관이 없습니다. 특히 Databricks CLI는 Skills를 만들 때 아주 유용합니다. CLI 명령어를 조합해서 Claude Code가 바로 실행할 수 있는 Skills를 쉽게 구성할 수 있거든요. 그리고 REST API를 활용하면 Custom MCP 서버도 만들 수 있습니다. 이번 핸즈온에서 MCP 서버도 만들어보고, Skills도 직접 만들어볼 겁니다.
