    Claude Code를 확장하는 방법은 크게 세 가지입니다. Skills: 슬래시 명령어로 호출하는 커스텀 워크플로우. Agents: 특정 역할에 특화된 AI 에이전트. Hooks: 특정 이벤트에 자동으로 트리거되는 스크립트. 이번 섹션에서는 Skills를 중심으로 다루고, Agents도 함께 살펴봅니다.

    Skills란?

    • SKILL.md 파일로 정의하는 커스텀 슬래시 커맨드
    • 프로젝트 내 .claude/skills/ 디렉토리에 위치
    • Claude Code가 특정 작업을 일관된 방식으로 수행하도록 가이드
    • 호출 방법: /skill-name 또는 자동 인식
    핵심 아이디어: "이 작업은 항상 이 절차대로 해줘"를 코드화한 것
    Skills의 핵심 개념을 설명합니다. SKILL.md는 마크다운 파일로, Claude Code에게 특정 작업의 수행 절차를 알려줍니다. 예를 들어 "슬라이드를 만들어줘"라고 할 때, 매번 스타일 가이드와 구조를 설명하는 대신, SKILL.md에 한번 정의해두면 일관된 결과물을 얻을 수 있습니다. 이것은 팀 전체가 동일한 품질의 결과물을 만들 수 있게 해주는 강력한 기능입니다.

    SKILL.md 파일 구조

    
# 스킬-이름 스킬

스킬에 대한 한줄 설명

## 사용법
/skill-name [파라미터1] [파라미터2]

## 입력
- **파라미터1**: 설명
- **파라미터2**: 설명

## 처리 절차
1. 레퍼런스 파일을 읽어 컨벤션 확인
2. 입력을 분석하여 구성 결정
3. 결과물 생성
4. 지정 경로에 저장

## 출력
- 생성되는 파일 경로와 설명

    사용법 섹션

    Skill을 호출하는 방법과 전달할 인자를 정의

    처리 절차 섹션

    가장 중요! 단계별 수행 절차를 순서대로 기술

    출력 형식 섹션

    최종 결과물의 형태와 포맷을 명시

    SKILL.md의 기본 구조입니다. 왼쪽에 전체 템플릿이 있고, 오른쪽에 각 섹션의 역할을 설명합니다. 사용법 섹션에서는 슬래시 명령과 인자를 정의합니다. 처리 절차 섹션은 가장 중요한 부분으로, AI가 어떤 순서로 작업을 수행할지 기술합니다. 출력 형식 섹션에서는 최종 결과물이 어떤 형태인지 명시합니다. 이 세 가지 섹션만 잘 작성하면 기본적인 Skill을 만들 수 있습니다.

    실제 예시: slide-deck 스킬

    이 프로젝트에서 실제 사용 중인 스킬

    
# slide-deck 스킬

reveal.js 기반 슬라이드 프레젠테이션을 생성합니다.

## 사용법
/slide-deck [섹션번호] [주제개요]

## 처리 절차
1. references/slide-conventions.md를 읽어
   슬라이드 스타일 가이드 확인
2. templates/reveal-template.html을 기반으로
   슬라이드 생성
3. 주제개요를 분석하여 적절한
   슬라이드 수와 구성 결정
4. 각 슬라이드에 스피커 노트 포함
5. sections/[번호]-[이름]/index.html에 저장

## 출력
- sections/XX-name/index.html
    이 프레젠테이션 자체가 slide-deck 스킬로 만들어진 것입니다. 하이라이트되는 부분을 따라가 보겠습니다. 먼저 스킬 이름과 설명이 있고, 사용법에서 섹션번호와 주제개요를 입력받습니다. 처리 절차가 핵심인데, 레퍼런스 파일을 먼저 읽고, 템플릿 기반으로 생성하며, 스피커 노트를 포함하는 등 구체적인 지시를 담고 있습니다. 이렇게 정의하면 누가 실행해도 동일한 스타일의 슬라이드가 생성됩니다.

    레퍼런스 파일

    Skills가 참조하는 상세 가이드라인

    .claude/skills/slide-deck/
├── SKILL.md                   ← 스킬 정의
└── references/
    └── slide-conventions.md   ← 상세 스타일 가이드

    conventions (컨벤션)

    코딩 스타일, 네이밍 규칙, 에러 처리 방식

    templates (템플릿)

    코드 생성의 기반이 되는 골격 코드

    examples (예시)

    완성된 코드 예시로 AI가 패턴 학습

    # api-conventions.md

## 응답 형식
{
  "status": "success" | "error",
  "data": { ... },
  "message": "설명"
}

## 네이밍 규칙
- 엔드포인트: kebab-case
- 함수명: snake_case
- 모델명: PascalCase

## 에러 처리
- HTTPException 사용
- 400, 404, 500 상태 코드
    레퍼런스 파일은 Skill의 품질을 결정짓는 핵심 요소입니다. SKILL.md는 큰 그림을 정의하고, references 디렉토리의 파일들이 세부 사항을 담당합니다. 세 가지 유형이 있습니다: Conventions는 팀의 코딩 컨벤션, Templates는 코드의 골격, Examples는 완성된 코드 예시입니다. 이렇게 분리하면 스킬의 처리 절차는 바꾸지 않으면서 스타일 가이드만 업데이트할 수 있습니다.

    Skill 설계 원칙

    1. 하나의 Skill = 하나의 명확한 목적

    Skill은 한 가지 작업만 수행해야 합니다. "API 엔드포인트 생성", "데이터 요약 보고서 작성" 등 목적이 명확해야 합니다.

    2. 입력 → 처리 → 출력 구조

    사용자 입력이 무엇이고, 어떤 처리를 거쳐, 어떤 결과물이 나오는지 명확히 정의합니다.

    3. 레퍼런스 파일로 품질 보장

    컨벤션, 템플릿, 예시 파일을 레퍼런스로 제공하면 AI가 일관된 품질의 코드를 생성합니다.

    좋은 Skill = 명확한 목적 + 구조화된 절차 + 레퍼런스 기반 품질

    Skill을 잘 설계하기 위한 세 가지 핵심 원칙입니다. 첫째, 하나의 Skill은 하나의 명확한 목적만 가져야 합니다. Unix 철학의 "한 가지를 잘 하라"와 같습니다. 둘째, 입력-처리-출력 구조를 명확히 정의해야 합니다. 함수를 설계하듯이 인터페이스가 명확해야 합니다. 셋째, 레퍼런스 파일을 적극 활용해야 합니다. 실제 컨벤션 문서나 예시 코드를 제공하는 것이 훨씬 정확한 결과를 냅니다.

    Skill 디렉토리 구조

    .claude/skills/
    ├── api-endpoint/
    ├   ├── SKILL.md ← 스킬 정의
    ├   └── references/
    ├       ├── api-conventions.md
    ├       ├── endpoint-template.py
    ├       └── example-endpoint.py
    ├── data-summary/
    ├   └── SKILL.md ← 간단한 스킬
    └── test-generator/
        ├── SKILL.md
        └── references/
            └── test-conventions.md

    핵심 포인트

    • .claude/skills/ 디렉토리에 배치
    • 각 Skill은 독립된 디렉토리
    • SKILL.md는 필수 파일
    • references/는 선택이지만 권장

    간단한 Skill: SKILL.md 하나만으로도 동작
    고품질 Skill: 레퍼런스 파일로 일관성 보장

    실제 프로젝트에서 Skill이 어떤 구조로 배치되는지 보겠습니다. .claude/skills 디렉토리 아래에 각 Skill별 디렉토리가 위치합니다. api-endpoint 스킬처럼 레퍼런스가 있는 스킬은 references 하위 디렉토리에 컨벤션, 템플릿, 예시 파일을 둡니다. data-summary 스킬처럼 간단한 경우에는 SKILL.md 하나만으로도 충분합니다. 핵심은 각 Skill이 독립된 디렉토리를 가진다는 것입니다.

    Custom Agents & Skills vs Agents

    Custom Agents

    • .claude/agents/ 디렉토리에 마크다운으로 정의
    • 특정 역할에 특화 (예: @researcher, @validator)
    • 도구 사용 권한과 제약 조건 명시
    • 호출: @agent-name
    SkillsAgents
    정의작업 절차역할과 권한
    파일SKILL.mdagent.md
    위치.claude/skills/.claude/agents/
    호출/skill-name@agent-name
    초점"무엇을, 어떻게""누가, 어떤 범위에서"
    핵심: Skills는 절차 중심, Agents는 역할 중심 — 다음 섹션에서 Agents를 자세히 다룹니다
    Agents는 Skills와는 다른 개념입니다. Skills가 작업의 절차를 정의한다면, Agents는 역할과 권한을 정의합니다. Skills는 "무엇을 어떻게 할 것인가"에 초점을 맞추고 슬래시 명령어로 호출합니다. Agents는 "누가 어떤 범위에서 할 것인가"에 초점을 맞추고 골뱅이로 호출합니다. 둘 다 마크다운 파일로 정의되며, .claude 디렉토리 아래에 위치합니다. 실무에서는 이 둘을 조합해서 사용합니다.

    Skill 체이닝 패턴

    Skill A의 출력을 Skill B의 입력으로 연결하여 복잡한 워크플로우를 구성합니다

    Skill A data-summary
    →
    출력 → 입력 요약 보고서
    →
    Skill B api-endpoint
    →
    Skill C test-generator

    체이닝 예시: 기능 개발 워크플로우

    1. /api-endpoint "사용자 프로필 조회 API"
   → 엔드포인트 코드 생성

2. /test-generator "생성된 엔드포인트에 대한 테스트"
   → 테스트 코드 자동 생성

3. /api-docs "프로필 조회 API 문서화"
   → API 문서 생성

    체이닝의 핵심: 각 Skill의 출력 형식이 다음 Skill의 입력 형식과 호환되어야 합니다

    Skill 체이닝은 여러 Skill을 연결하여 복잡한 워크플로우를 구성하는 패턴입니다. 예를 들어, 기능 개발 워크플로우를 생각해봅시다. 먼저 api-endpoint 스킬로 엔드포인트 코드를 생성하고, 그 결과물을 기반으로 test-generator 스킬로 테스트 코드를 만들고, 마지막으로 api-docs 스킬로 API 문서를 생성합니다. 체이닝이 잘 동작하려면 각 Skill의 출력 형식이 다음 Skill의 입력으로 자연스럽게 연결되어야 합니다.

    3. /api-endpoint를 실행하여 생성된 코드를 테스트합니다

    > exercise_02_skill_with_refs/SKILL.md에 api-endpoint 스킬을 작성해줘

    체크리스트

    • SKILL.md에 사용법이 명확히 정의되어 있는가?
    • 처리 절차가 단계별로 기술되어 있는가?
    • 출력 형식이 명시되어 있는가?
    • 레퍼런스 파일이 실제로 참조되고 있는가?
    • Skill을 실행했을 때 일관된 결과가 나오는가?
    이제 실습 시간입니다. 두 가지 Exercise를 진행합니다. 워크플로우는 동일합니다: 먼저 SKILL.md 템플릿의 TODO 항목을 확인하고, Claude Code에게 작성을 요청한 뒤, 실제로 실행하여 테스트합니다. Exercise 1은 간단한 Skill 만들기입니다. 약 10분 정도 소요됩니다. Exercise 2는 레퍼런스 파일이 있는 Skill입니다. 약 15분 정도 소요됩니다. 체크리스트를 활용하여 Claude Code가 작성한 Skill이 올바르게 구성되었는지 검토해보세요.

    정리

    핵심 정리

    • Skills — SKILL.md로 정의하는 커스텀 슬래시 커맨드. 작업 절차를 코드화
    • Skill 설계 원칙 — 하나의 목적, 입력-처리-출력 구조, 레퍼런스 기반 품질
    • 레퍼런스 파일 — conventions, templates, examples로 일관성 보장
    • Skill 체이닝 — 출력-입력 연결로 복잡한 워크플로우 구성
    • Agents — 역할과 권한 기반. Skills와 조합하여 사용
    정리하겠습니다. Skills는 작업 절차를 SKILL.md에 정의하여 슬래시 명령어로 호출합니다. 좋은 Skill은 명확한 목적, 구조화된 절차, 레퍼런스 기반 품질 세 가지를 갖춥니다. 레퍼런스 파일로 일관성을 보장하고, 체이닝으로 워크플로우를 구성합니다. Agents는 역할 기반으로 Skills와 상호 보완적입니다.
