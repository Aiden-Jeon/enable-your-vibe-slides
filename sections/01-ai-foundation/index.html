    AI Fluency란?

    "

    AI와 효과적으로, 효율적으로, 윤리적으로, 안전하게 협업할 수 있는 능력

    — Anthropic Academy, AI Fluency: Framework & Foundations

    • 도구 사용 이상의 사고방식 전환
    • AI 협업의 전체 생명주기를 체계적으로 관리
    • Anthropic이 제안하는 4D Framework로 구체화
    AI Fluency라고 하는 거는 AI와 효과적으로, 효율적으로, 윤리적으로, 안전하게 협업할 수 있는 능력입니다. 여기 링크에 들어가 보시면 Anthropic에서 나오는 코스에 대해서 설명이 있는데, 혹시 제가 설명하는 거 외에도 조금 더 관심이 있다면 이 세션 한 번 들어보시면 좋을 것 같습니다. 아무래도 여러 가지 도구들이 있지만, 사실 내가 어떻게 AI와 협업을 해야 되는지, AI는 뭘 할 수 있고 나는 어떤 걸 해야 되는지에 대해서 개념을 확고하게 하기 어렵습니다. 그래서 Anthropic에서 제안한 게 이 AI Fluency라는 개념이라고 봐주시면 될 것 같고요. AI 협업의 전체적인 생명 주기를 이 프레임워크에 맞춰서 관리하는 것을 얘기합니다.

    4D Framework

    Delegation (위임)

    AI에게 무엇을 맡기고, 무엇은 직접 할 것인가?

    Description (기술)

    AI에게 의도를 어떻게 명확히 전달할 것인가?

    Discernment (분별)

    AI의 결과물을 어떻게 평가할 것인가?

    Diligence (책임)

    AI를 윤리적이고 안전하게 사용하고 있는가?

    그 프레임워크를 4D 프레임워크라고 하는데, 여기서 전부 다 설명을 드릴 수는 없을 것 같고 몇 가지만 좀 살펴볼 겁니다. 4D라는 거는 첫째, Delegation — AI에게 어떤 거를 맡기고 내가 어떤 걸 할 것인지에 대해서 우선 설정을 하고, 둘째, Description — AI에게 내가 맡기고 싶은 일을 어떻게 명확하게 전달할 건지에 대한 부분. 셋째, Discernment — 시켰으면 우리가 어떻게 평가할 건지가 사실 되게 중요합니다. AI가 만든 결과물을 어떻게 평가할 것인가. 예를 들어 바이브 코딩을 할 거기 때문에 생성 결과물이 테스트 코드를 통과했다든지, 혹은 원하는 앱을 만든다면 그 앱이 원하는 기술 명세서를 전부 다 충족했는지 이런 것들이 평가 요소가 될 수 있을 겁니다. 마지막으로 Anthropic 같은 경우에는 되게 강조하는 것이 Safe한 AI입니다. Diligence라고 하는 책임 부분이 있는데 사실 이거는 조금 여기에서는 다루지 않을 내용입니다.

    Delegation 깊이 보기

    누가 무엇을 할 것인가를 사려 깊게 결정하는 것

    나 혼자 • AI 혼자 • 함께 — 그리고 어떻게 분배할 것인가

    1. Problem Awareness

    문제 인식 —
    목표와 성공 기준을 먼저 명확히 한다

    2. Platform Awareness

    플랫폼 인식 —
    AI 시스템의 강점과 한계를 이해한다

    3. Task Delegation

    작업 분배 —
    사람 vs AI vs 협업, 전략적으로 배분한다

    우선 Delegation에 대해서 한 번 볼 겁니다. Delegation은 누가 무엇을 할 것인가를 사려 깊게 결정하는 것, 나 혼자 할 것, AI가 혼자 할 것, 그리고 우리가 같이 할 것에 대한 거를 어떤 식으로 분배할 건지에 대한 내용입니다.

    Delegation 3 Pillars

    1. Problem Awareness — 무엇을 하려는가?

    • "성공"은 어떤 모습인가? 어떤 사고와 작업이 필요한가?
    • AI fluency는 본인의 전문성에서 시작한다

    2. Platform Awareness — AI는 무엇을 잘하는가?

    • 어떤 모델이 적합한가? 속도 vs 창의성 vs 깊이 vs 정확성
    • 도구별 강점과 한계를 파악한다

    3. Task Delegation — 자, 이제 계획을 세우자

    • 자동화할 수 있는 것은? 협업이 더 가치 있는 곳은? 사람만 해야 할 것은?
    • 목표: 모든 것을 자동화하는 것이 아니라 최적의 인간-AI 파트너십
    여기서 세 가지로 볼 수 있는데요. 우선은 Problem Awareness, 문제 인식 — 우리가 풀고자 하는 목표, 그리고 어떻게 해야 이게 성공했는지에 대해서 기준을 명확하게 해야 됩니다. 그리고 Platform Awareness가 중요합니다. 사실 되게 여러 가지 툴들이 있잖아요. 코딩을 하는데 Claude 자체를 쓸 수도 있고, GPT를 쓸 수도 있고, Gemini 같은 툴들이 있는데 각각의 시스템들이 어떤 부분에서 강점을 가지고 있고, 어떤 부분에서 한계를 가지고 있는지 명확하게 인식을 하고 있어야 됩니다. 그래서 맡기는 일, 위임하고자 하는 일이 그 툴에서 잘 되는지를 알고 있어야 잘 줄 수 있고요. 그리고 앞서 말씀드린 것처럼 사람이 잘하는 부분, AI가 잘하는 부분, 그리고 우리가 협업해야 되는 부분들을 잘 배분하는 게 중요합니다.

    Delegation 실습 실습

    databricks console → Playground → Claude Sonnet 4.5 선택

    프롬프트 템플릿:

    안녕 Claude, [작업 내용]을 준비하고 있는데,
어떤 부분을 당신 같은 AI에게 맡기고
어떤 부분은 직접 해야 할지
위임 계획을 함께 논의하고 싶어. 도와줄 수 있겠니?

    예시 (데이터 분석 에이전트 PoC):

    안녕 Claude, 데이터 분석 에이전트 개발 PoC를 준비하고 있는데,
어떤 부분을 당신 같은 AI에게 맡기고
어떤 부분은 직접 해야 할지
위임 계획을 함께 논의하고 싶어. 도와줄 수 있겠니?

이번 프로젝트의 최종 목표는:
유저가 챗봇에게 업무 관련 질문을 하면,
챗봇이 관련 데이터셋을 추출하는 쿼리를 생성하고,
추출된 데이터에서 인사이트를 제시하는 것이야.
    예를 들어서 한 번 실습을 해볼 예정인데, Databricks 콘솔에 들어가셔서 Playground에서 Claude Sonnet에 한 번 이 내용을 질문할 겁니다. 예를 들어서 데이터 분석 에이전트 PoC를 만들 거면 제가 만들어 놓은 이 부분이 있는데 이거를 한 번 같이 실습을 해보도록 하겠습니다.

    Delegation 실습 예시

    Playground에서 Claude와 위임 계획을 논의하는 화면

    이렇게 들어오셔서 최근에 Claude Sonnet 4.6이 나와서 4.6을 한번 사용해볼까요? 실행해주시면 이런 식으로 우리가 어떻게 위임을 할 수 있는지에 대한 부분들에 대해서 설명들이 나오고 있습니다. 아 내가 어떤 부분들을 준비를 해야 되는구나에 대한 감을 얻을 수 있는 거고. 근데 재밌는 부분이 뭐냐면, Claude 같은 경우에는 이 Delegation이라는 것들을 항상 강조하기 때문에 모델에서도 이런 부분들을 반영을 많이 해놨습니다. 근데 예를 들어서 ChatGPT를 똑같이 사용하면 사실 GPT 같은 경우에는 이게 잘 안 됩니다. Gemini로 가게 되면 더 안 되고요. 이게 모든 모델들마다 되는 건 아니고, Anthropic에서 만든 거기 때문에 Anthropic에서는 이 부분들이 되게 잘 되어있고 나머지들은 조금 애매하게 나오는 걸 볼 수 있죠. GPT 같은 경우에는 5.2로 넘어오면서 이 부분이 좀 잘 나오게 되었고, Gemini 같은 경우에는 되게 잘 안 되는 것을 보실 수가 있습니다. 그래서 보통 많이 사용하는 게 뭐냐면, Anthropic이 전체적인 메타 플래닝을 진행하는 에이전트고요. 여기에서 자료 조사를 수행할 거다 그러면 Gemini한테 던져서 결과들을 받아와서 다시 Claude한테 전달해서 새로운 인사이트를 얻을 수도 있고, Claude Code를 GPT한테 개발을 시키고 나온 결과물들을 다시 줘서 메타에 대한 부분들을 Anthropic을 사용하는 게 경험적으로 가장 좋은 부분 중 하나였습니다.

    Description 깊이 보기

    AI와 효과적으로 소통하는 기술 — 단순한 프롬프트 작성이 아니다

    작업을 설명하고, 질문하고, 맥락을 제공하고, 상호작용을 안내한다

    1. Product Description

    무엇을 만들 것인가 —
    원하는 결과물을 명확히 정의한다

    2. Process Description

    어떻게 접근할 것인가 —
    AI의 사고 과정을 안내한다

    3. Performance Description

    어떻게 행동할 것인가 —
    AI의 행동 방식을 정의한다

    다음으로는 Description인데요. Description은 AI와 효과적으로 소통하는 기술입니다. 그러기 위해서는 이 세 가지가 필요합니다. Product Description, Process Description, Performance Description이 필요한데요.

    Description 3 Pillars

    1. Product Description — 무엇을 원하는가?

    • 원하는 것을 명확히 기술: 맥락, 형식, 대상, 스타일, 제약 조건
    • AI가 추측하지 않도록 — 필요한 정보를 직접 제공한다

    2. Process Description — 어떻게 접근할 것인가?

    • "어떻게"가 "무엇"보다 중요할 수 있다
    • 구체적 데이터, 핵심 작업, 선호하는 순서 등 사고 과정을 안내

    3. Performance Description — 어떻게 행동할 것인가?

    • AI는 맥락에 따라 다르게 행동하는 인터랙티브 시스템
    • 간결하게 vs 자세하게, 도전적으로 vs 지지적으로 — 원하는 행동 방식을 정의
    Product Description은 내가 무엇을 원하는지 명시적으로 쓰는 겁니다. "이런 걸 해줘"라는 식으로 그냥 AI가 추측하도록 하게 하면 사실 성능이 안 좋게 나오는 경우들이 거의 대부분입니다. 그래서 이거를 명시해줘야 되고요. Process Description은 이 문제를 어떤 식으로 풀지에 대해서 사고하는 과정을 제공해드리는 거죠. 물론 이 부분의 경우에는 모델들을 디벨롭하면서 자체적으로 thinking이라든지 여러 가지 reasoning들이 들어가면서 어느 정도 유사하긴 하지만, 사실 사용자가 원하는 방식을 명시적으로 주는 게 좋습니다. 만약에 이게 명확하지 않다면 이 과정마저도 AI와 티키타카를 할 수가 있게 되는 거죠. Performance Description은 예를 들어서 디테일하게 볼 건지 아니면 간결하게 볼 건지, 어떤 톤으로 대답할 건지 이런 부분에 대한 거를 시스템으로 줄 수도 있고 아니면 그 세션 자체에서도 줄 수도 있고요.

    Description 실습 실습

    Playground에서 부족한 프롬프트를 개선하는 연습

    프롬프트:

    목표: 프롬프트 Description에 대한 연습 경험

Context:
- 다음 항목들이 부족한 프롬프트를 작성해서 개선하는 것을 연습하려고 해
  - Product description
  - Process description
  - Performance description

Desired output:
위의 컨텍스트 항목이 부족한 프롬프트

Process:
1. Claude가 부족한 프롬프트 예시를 작성한다.
2. 내가 부족한 부분을 개선해서 답변한다.
3. Claude가 개선한 부분에 대해서 평가해준다.

    Claude가 만든 부족한 프롬프트를 Product • Process • Performance 관점에서 개선해보세요

    한 번 실습을 해볼 건데 이거를 다시 Playground로 넘어와서 한 번 실행을 시켜보도록 하겠습니다. 이게 그냥 연습하는 방법이라고 봐주시면 돼요. 여기에서 예를 들어서 "마케팅 이메일 써줘"라고 하는 부족한 걸 주면, 아까 Product Description, Process Description, Performance Description이 필요하다고 했으니까 제가 좀 써보도록 하겠습니다. 예를 들어 "Databricks AI Day에 대한 초대 메일을 작성해줘" 이렇게 쓰고, Process Description으로 "간단한 데다가 해킹할 수 있는 제목을 갖고 있어야 해", Performance Description으로 "한국어고 허세감이 없어야 해" 이렇게 넣어볼 수 있습니다. 아까 말씀드린 것처럼 내가 Description을 잘 했는지 안 했는지에 대해서 의문이 들면 사실 이 과정도 AI와 대화할 수 있는 거죠. 내가 이렇게 썼던 프롬프트가 좋은지 안 좋은지에 대해서도 얘가 평가를 해주고요. 이런 식으로 티키타카를 해서 원하는 결과를 한 번에 얻을 수 있는 확률이 올라가게 되는 거죠.

    4D → Vibe Coding

    "

    There's a new kind of coding I call 'vibe coding' where you fully give in to the vibes, embrace exponentials, and forget that the code even exists.

    — Andrej Karpathy, OpenAI 창립 멤버, 2025년 2월

    4D Framework이 Vibe Coding에서 의미하는 것

    Delegation → 코드 작성을 AI에게 위임
    Description → 자연어로 의도를 전달
    Discernment → AI 생성 코드의 품질 검증
    Diligence → 보안, 라이선스, 윤리 확인

    역할 전환: 코더 → 오케스트레이터

    4D에서 Vibe Coding으로 얘기해보자면, Vibe Coding 같은 경우에는 Andrej Karpathy라는 사람이 "바이브로만 코딩을 하는 것 같다"라고 처음 언급했던 게 확산이 되면서 자연어로만 코딩을 하는 전반적인 과정들을 Vibe Coding이라고 얘기를 하고 있습니다. 4D Framework에 맞춰서 다시 Vibe Coding을 보면, 메타에서 어떤 걸 만들지에 대해서 명세서를 썼고(Delegation), 어떤 코드를 AI에게 주는 거고 기술 명세서라든지 만들어야 될 것들을 줄 수 있고(Description), 만들어진 게 테스트 코드를 쓰거나 의도한 대로 동작하는지 볼 수가 있습니다(Discernment). 이게 역할의 전환으로 보자면 개발을 하는 사람에서 여러 가지 에이전트들을 오케스트레이팅 해주는 오케스트레이터 역할로 변화된 거죠.

    AI 코딩 도구의 발전

    도구 방식 특징 4D 관점
    Copilot 자동 완성 에디터 내 인라인 제안, Tab으로 수락 부분 위임, 수동 분별
    Cursor 채팅 + 편집 멀티 파일 편집, Composer 모드 대화형 기술, 반자동 분별
    Claude Code 에이전틱 터미널 기반, 자율적 도구 사용, MCP 확장 전면 위임, 체계적 책임

    자동 완성 → 대화형 편집 → 자율적 에이전트

    에이전틱 코딩 = AI가 도구를 사용하여 자율적으로 계획 → 실행 → 검증

    발전으로 보자면 처음에는 Copilot이 자동 완성을 도와주다가, Cursor 같은 경우에는 에디터에 이런 에이전트를 붙였었는데, 사실 Claude Code 같은 경우에는 Claude에 있었던 해커톤에서 나왔었던 프로젝트인데 자연어로 코딩을 하게 되면 코드 화면을 볼 필요가 없다라고 하는 게 어떻게 보면 시작점이었습니다. 여러 가지 도구들이 나왔었지만 사실 지금에 와서는 Claude Code가 결국 승자인 거죠. 왜냐면 얘가 작성해주는 것들에 대한 양들을 사실 다 보지 않게 되는 경우들이 많이 있고, 그냥 터미널에서 자연어로만 명령을 시키고 테스트만 빠르게 실행시키면서 결과들을 보는 식으로 개발을 하게 됐기 때문에 CLI에서 에이전틱이라고 하는 방법으로 하는 것들이 되게 표준이 된 상황입니다.
