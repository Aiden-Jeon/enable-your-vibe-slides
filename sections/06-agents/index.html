    • agent.md 파일로 정의하는 역할 기반 AI 에이전트
    • 프로젝트 내 .claude/agents/ 디렉토리에 위치

# Claude Code에서 에이전트 호출
> @researcher 최신 MCP 프로토콜 변경사항을 조사해줘
> @validator 이 슬라이드 섹션을 검증해줘
    Custom Agents의 핵심 개념을 설명합니다. agent.md는 마크다운 파일로, Claude Code에게 특정 역할의 전문가처럼 행동하도록 지시합니다. Skills가 작업 절차를 정의한다면, Agents는 역할과 관점을 정의합니다. 예를 들어 researcher 에이전트는 조사 전문가로, validator 에이전트는 검증 전문가로 동작합니다. 골뱅이(@) 기호로 호출하면 해당 역할에 맞는 응답을 받을 수 있습니다.

    agent.md 파일 구조

    
# 에이전트 이름

에이전트에 대한 한줄 설명

## 역할 (Role)
당신은 [전문 분야] 전문가입니다.
[역할에 대한 상세 설명]

## 제약 조건 (Constraints)
- 조사 범위 제한
- 사용 가능한 도구 명시
- 금지 사항

## 출력 형식 (Output Format)
결과물의 형태를 정의합니다:
- 마크다운 보고서
- 체크리스트

    역할 섹션

    에이전트의 전문성과 관점을 정의. "당신은 ~입니다"로 시작

    제약 조건 섹션

    에이전트가 하지 말아야 할 것, 범위 제한, 사용 도구 명시

    출력 형식 섹션

    결과물의 구조와 포맷을 미리 정의하여 일관성 보장

    agent.md의 기본 구조입니다. 역할 섹션에서는 에이전트의 전문성을 정의합니다. "당신은 ~ 전문가입니다"로 시작하는 것이 좋습니다. 제약 조건 섹션에서는 에이전트가 하지 말아야 할 것과 범위를 명시합니다. 출력 형식 섹션에서는 결과물이 어떤 형태여야 하는지 정의합니다. Skills의 SKILL.md와 비교하면, Skills는 처리 절차가 핵심이고, Agents는 역할과 제약 조건이 핵심입니다.

    실제 예시: researcher 에이전트

    이 프로젝트에서 실제 사용 중인 에이전트

    
# researcher

교육 콘텐츠 리서치 전문 에이전트

## 역할
당신은 AI/ML 및 개발 도구 분야의 리서치 전문가입니다.
교육 자료 제작에 필요한 최신 정보를 수집하고,
정확하고 실용적인 내용을 정리합니다.

## 제약 조건
- 공식 문서와 신뢰할 수 있는 소스만 참조
- 추측이나 불확실한 정보는 명시적으로 표시
- 코드 예시는 반드시 동작 가능한 것만 포함
- 한국어로 결과물 작성

## 출력 형식
### 조사 결과
- **주제**: [조사 주제]
- **핵심 내용**: 불릿 포인트로 정리
- **참고 소스**: URL 목록
- **교육 자료 반영 포인트**: 슬라이드에 활용할 내용
    이 프로젝트에서 실제로 사용 중인 researcher 에이전트입니다. 역할에서 AI/ML 및 개발 도구 분야의 리서치 전문가로 정의했습니다. 제약 조건에서 공식 문서만 참조하고, 불확실한 정보는 명시하도록 했습니다. 출력 형식에서 주제, 핵심 내용, 참고 소스, 교육 자료 반영 포인트를 구조화했습니다. 이런 구조 덕분에 누가 @researcher를 호출해도 동일한 형식의 조사 결과를 받을 수 있습니다.

    실제 예시: validator 에이전트

    
# validator

슬라이드 및 코드 품질 검증 에이전트

## 역할
당신은 교육 콘텐츠 품질 검증 전문가입니다.
슬라이드의 정확성, 일관성, 완성도를
체계적으로 검증합니다.

## 체크리스트
- [ ] 섹션 번호와 제목이 section.yaml과 일치
- [ ] 모든 슬라이드에 스피커 노트 포함
- [ ] 코드 블록에 language 속성 지정
- [ ] 다크 테마 색상 컨벤션 준수
- [ ] 이전/다음 섹션 참조가 올바른지
- [ ] reveal.js CDN 링크가 최신 버전인지

## 출력 형식
### 검증 결과
- **통과 항목**: [체크 표시 목록]
- **수정 필요**: [이슈와 수정 방안]
- **경고**: [주의사항]
- **종합 평가**: PASS / FAIL (사유)
    validator 에이전트는 품질 검증에 특화되어 있습니다. 역할에서 교육 콘텐츠 품질 검증 전문가로 정의했습니다. 체크리스트 섹션이 핵심인데, 검증해야 할 항목을 구체적으로 나열합니다. 출력 형식에서 통과 항목, 수정 필요, 경고, 종합 평가를 구조화했습니다. 이 에이전트를 사용하면 슬라이드를 만든 후 일관된 기준으로 품질을 검증할 수 있습니다.

    Skills vs Agents 비교

    SkillsAgents
    정의작업 절차 (How)역할과 관점 (Who)
    파일SKILL.mdagent.md
    위치.claude/skills/.claude/agents/
    호출/skill-name@agent-name
    초점"무엇을, 어떻게""누가, 어떤 범위에서"
    예시/slide-deck, /api-endpoint@researcher, @validator
    선택 가이드:
    반복적인 작업 절차가 있다 → Skill | 특정 전문가 관점이 필요하다 → Agent
    Skills와 Agents의 핵심 차이를 비교합니다. Skills는 작업의 절차, 즉 How를 정의합니다. "이 작업은 이 순서대로 해줘." Agents는 역할과 관점, 즉 Who를 정의합니다. "이 전문가처럼 행동해줘." 선택 기준은 간단합니다. 동일한 작업을 반복적으로 수행해야 한다면 Skill을 만듭니다. 특정 전문가의 관점이 필요하다면 Agent를 만듭니다. 실무에서는 둘을 조합해서 사용합니다. 예를 들어 @researcher에게 조사를 요청한 후, /slide-deck으로 슬라이드를 생성할 수 있습니다.

    Agent 설계 원칙

    1. 명확한 역할 정의

    "당신은 ~ 전문가입니다"로 시작. 역할이 모호하면 결과도 모호해집니다.

    2. 적절한 제약 조건

    하지 말아야 할 것을 명시. 범위가 넓을수록 제약 조건이 중요합니다.

    3. 구조화된 출력 형식

    결과물의 형태를 미리 정의하면 일관된 품질을 보장할 수 있습니다.

    4. 단일 책임 원칙

    하나의 Agent는 하나의 역할에 집중. 여러 역할이 필요하면 여러 Agent를 만듭니다.

    좋은 Agent를 설계하기 위한 네 가지 원칙입니다. 첫째, 역할을 명확하게 정의합니다. 모호한 역할은 모호한 결과를 낳습니다. 둘째, 적절한 제약 조건을 둡니다. 특히 범위가 넓은 에이전트일수록 제약이 중요합니다. 셋째, 출력 형식을 미리 정의합니다. 매번 다른 형식으로 결과가 나오면 활용하기 어렵습니다. 넷째, 단일 책임 원칙을 따릅니다. 리서치와 검증을 하나의 Agent에 넣지 마세요.

    Agent 디렉토리 구조

    .claude/
    ├── agents/ ← Agent 정의
    ├   ├── researcher.md ← 리서치 전문가
    ├   └── validator.md ← 검증 전문가
    ├── skills/ ← Skill 정의
    ├   ├── slide-deck/
    ├   ├── hands-on-code/
    ├   └── section-builder/
    └── CLAUDE.md ← 프로젝트 컨텍스트

    핵심 포인트

    • .claude/agents/ 디렉토리에 배치
    • 각 Agent는 하나의 .md 파일
    • 파일명 = 에이전트 호출명
    • Skills보다 구조가 단순

    Skills: 디렉토리 + SKILL.md + references/
    Agents: agent-name.md 파일 하나로 완성

    실제 프로젝트에서 Agent가 어떻게 배치되는지 보겠습니다. .claude 디렉토리 아래에 agents 폴더가 있고, 각 에이전트는 하나의 마크다운 파일입니다. Skills와 비교하면 구조가 훨씬 단순합니다. Skills는 디렉토리 안에 SKILL.md와 references를 가지지만, Agents는 .md 파일 하나로 완성됩니다. 파일명이 곧 호출명이 됩니다. researcher.md → @researcher로 호출합니다.

    체크리스트

    • 역할이 명확하게 정의되어 있는가?
    • 제약 조건이 적절히 설정되어 있는가?
    • 출력 형식이 구조화되어 있는가?
    • 단일 책임 원칙을 따르는가?
    • 에이전트를 호출했을 때 기대한 역할로 동작하는가?
    실습 시간입니다. 두 가지 Exercise를 진행합니다. Exercise 1은 간단한 code-reviewer 에이전트를 만드는 것입니다. 템플릿 파일을 확인한 후 Claude Code에게 작성을 요청하고, 실제로 호출하여 테스트합니다. Exercise 2는 좀 더 자유롭게, 팀에서 실제로 필요한 역할의 에이전트를 만들어봅니다. data-analyst, sql-optimizer, doc-writer 등 자신의 업무에 맞는 에이전트를 설계해보세요. 체크리스트를 참고하여 품질을 검증합니다.

    정리

    핵심 정리

    • Custom Agents — agent.md로 정의하는 역할 기반 AI 에이전트
    • 핵심 구조 — 역할 + 제약 조건 + 출력 형식
    • Skills vs Agents — 절차 중심 vs 역할 중심, 실무에서는 조합 사용
    • 설계 원칙 — 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임
    정리하겠습니다. Custom Agents는 agent.md 파일 하나로 역할 기반 AI 에이전트를 정의합니다. 핵심 구조는 역할, 제약 조건, 출력 형식 세 가지입니다. Skills는 절차 중심, Agents는 역할 중심으로, 실무에서는 둘을 조합해서 사용합니다. 좋은 Agent는 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임 원칙을 갖춥니다.
