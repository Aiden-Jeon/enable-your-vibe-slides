# Claude Code에서 에이전트 호출
> @genie-space-designer catalog.schema의 테이블을 분석해서 Space를 설계해줘
> @genie-instructor 이 테이블 구성에 맞는 instructions를 작성해줘
    Custom Agents의 핵심 개념을 설명합니다. agent.md는 마크다운 파일로, Claude Code에게 특정 역할의 전문가처럼 행동하도록 지시합니다. Skills가 작업 절차를 정의한다면, Agents는 역할과 관점을 정의합니다. 예를 들어 genie-space-designer 에이전트는 Space 설계 전문가로, genie-instructor 에이전트는 instructions 작성 전문가로 동작합니다. 골뱅이(@) 기호로 호출하면 해당 역할에 맞는 응답을 받을 수 있습니다.

    agent.md 파일 구조

    
# 에이전트 이름

에이전트에 대한 한줄 설명

## 역할 (Role)
당신은 [전문 분야] 전문가입니다.
[역할에 대한 상세 설명]

## 제약 조건 (Constraints)
- 조사 범위 제한
- 사용 가능한 도구 명시
- 금지 사항

## 출력 형식 (Output Format)
결과물의 형태를 정의합니다:
- 마크다운 보고서
- 체크리스트

    역할 섹션

    에이전트의 전문성과 관점을 정의. "당신은 ~입니다"로 시작

    제약 조건 섹션

    에이전트가 하지 말아야 할 것, 범위 제한, 사용 도구 명시

    출력 형식 섹션

    결과물의 구조와 포맷을 미리 정의하여 일관성 보장

    agent.md의 기본 구조입니다. 역할 섹션에서는 에이전트의 전문성을 정의합니다. "당신은 ~ 전문가입니다"로 시작하는 것이 좋습니다. 제약 조건 섹션에서는 에이전트가 하지 말아야 할 것과 범위를 명시합니다. 출력 형식 섹션에서는 결과물이 어떤 형태여야 하는지 정의합니다. Skills의 SKILL.md와 비교하면, Skills는 처리 절차가 핵심이고, Agents는 역할과 제약 조건이 핵심입니다.

    실제 예시: genie-space-designer 에이전트

    Genie Space 설계 전문 에이전트

    
# genie-space-designer
Genie Space 설계 전문 에이전트

## 역할
당신은 Databricks Genie Space 아키텍트입니다.
테이블 스키마를 분석하고 최적의 Space 구성을 설계합니다.

## 제약 조건
- 테이블 스키마 분석만 수행 (Space 생성 금지)
- 테이블은 5개 이하로 추천
- 한국어로 결과물 작성

## 출력 형식
- **분석 대상 테이블**: 스키마 요약
- **추천 테이블 구성**: 선정 이유 포함
- **Sample Questions**: 5개 이상
- **Space 방향성**: 분석 관점 제안
    Genie Space 설계에 특화된 genie-space-designer 에이전트입니다. 역할에서 Genie Space 아키텍트로 정의하여, 테이블 스키마를 분석하고 최적의 구성을 설계합니다. 제약 조건에서 스키마 분석만 수행하고, 실제 Space 생성은 하지 않도록 제한했습니다. 출력 형식에서 분석 대상 테이블, 추천 구성, Sample Questions, Space 방향성을 구조화했습니다. 이 에이전트는 Section 04에서 만든 Genie MCP의 첫 번째 활용 단계입니다.

    실제 예시: genie-instructor 에이전트

# genie-instructor
Genie Space instructions 작성 전문 에이전트

## 역할
당신은 serialized_space instructions 블록 전문가입니다.
text_instructions, example_question_sqls,
sql_snippets를 작성합니다.

## 제약 조건
- instructions 블록 작성만 수행
- example_question_sqls 3개 이상 포함
- sql_snippets의 expressions, measures, filters 작성
- Space 생성이나 질의는 수행하지 않음

## 출력 형식
- **text_instructions**: 비즈니스 컨텍스트 설명
- **example_question_sqls**: 질문-SQL 쌍
- **sql_snippets**: expressions, measures, filters
- **join_specs**: 테이블 간 조인 조건
    genie-instructor 에이전트는 Genie Space instructions 작성에 특화되어 있습니다. 역할에서 serialized_space의 instructions 블록 전문가로 정의했습니다. 제약 조건에서 instructions 블록 작성만 수행하고, Space 생성이나 질의는 하지 않도록 제한했습니다. 출력 형식에서 text_instructions, example_question_sqls, sql_snippets, join_specs를 구조화했습니다. designer가 설계한 Space 구성을 기반으로, instructor가 실제 instructions을 작성하는 역할 분담입니다.

# 라이프사이클 순서대로 호출
> @genie-space-designer catalog.schema의 테이블을 분석해서 Space를 설계해줘
> @genie-instructor designer 결과를 기반으로 instructions를 작성해줘
> @genie-tester Space에 sample questions를 질의하고 응답 품질을 평가해줘

    스토리라인: Section 04 MCP Tools → Section 05 Skill → Section 06 Agents

    세 에이전트가 Genie Space 라이프사이클의 각 단계를 담당합니다. Phase 1: designer가 테이블 스키마를 분석하고 최적의 Space 구성을 설계합니다. Phase 2: instructor가 designer의 결과를 기반으로 text_instructions, example_sqls, sql_snippets를 작성합니다. Phase 3: tester가 완성된 Space에 ask_genie로 질의하고, 응답 품질을 평가합니다. genie-tester 에이전트는 ask_genie 도구로 sample questions를 질의하고, 응답의 정확성과 완전성을 평가하는 역할입니다. 이렇게 Section 04에서 만든 MCP 도구, Section 05에서 만든 Skill, Section 06에서 만든 Agents가 하나의 워크플로우로 연결됩니다.

    Skills vs Agents 비교

    SkillsAgents
    정의작업 절차 (How)역할과 관점 (Who)
    파일SKILL.mdagent.md
    위치.claude/skills/.claude/agents/
    호출/skill-name@agent-name
    초점"무엇을, 어떻게""누가, 어떤 범위에서"
    예시/genie-space, /data-summary@genie-space-designer, @genie-instructor
    선택 가이드:
    반복적인 작업 절차가 있다 → Skill | 특정 전문가 관점이 필요하다 → Agent
    Skills와 Agents의 핵심 차이를 비교합니다. Skills는 작업의 절차, 즉 How를 정의합니다. "이 작업은 이 순서대로 해줘." Agents는 역할과 관점, 즉 Who를 정의합니다. "이 전문가처럼 행동해줘." 선택 기준은 간단합니다. 동일한 작업을 반복적으로 수행해야 한다면 Skill을 만듭니다. 특정 전문가의 관점이 필요하다면 Agent를 만듭니다. 실무에서는 둘을 조합해서 사용합니다. 예를 들어 @genie-space-designer에게 설계를 요청한 후, @genie-instructor로 instructions를 작성할 수 있습니다.

    Agent 설계 원칙

    1. 명확한 역할 정의

    "당신은 ~ 전문가입니다"로 시작. 역할이 모호하면 결과도 모호해집니다.

    2. 적절한 제약 조건

    하지 말아야 할 것을 명시. 범위가 넓을수록 제약 조건이 중요합니다.

    3. 구조화된 출력 형식

    결과물의 형태를 미리 정의하면 일관된 품질을 보장할 수 있습니다.

    4. 단일 책임 원칙

    하나의 Agent는 하나의 역할에 집중. 여러 역할이 필요하면 여러 Agent를 만듭니다.

    좋은 Agent를 설계하기 위한 네 가지 원칙입니다. 첫째, 역할을 명확하게 정의합니다. 모호한 역할은 모호한 결과를 낳습니다. 둘째, 적절한 제약 조건을 둡니다. 특히 범위가 넓은 에이전트일수록 제약이 중요합니다. 셋째, 출력 형식을 미리 정의합니다. 매번 다른 형식으로 결과가 나오면 활용하기 어렵습니다. 넷째, 단일 책임 원칙을 따릅니다. 리서치와 검증을 하나의 Agent에 넣지 마세요.

    핵심 포인트

    • .claude/agents/ 디렉토리에 배치
    • 각 Agent는 하나의 .md 파일
    • 파일명 = 에이전트 호출명
    • Skills보다 구조가 단순

    Skills: 디렉토리 + SKILL.md + references/
    Agents: agent-name.md 파일 하나로 완성

    실제 프로젝트에서 Agent가 어떻게 배치되는지 보겠습니다. .claude 디렉토리 아래에 agents 폴더가 있고, 각 에이전트는 하나의 마크다운 파일입니다. Skills와 비교하면 구조가 훨씬 단순합니다. Skills는 디렉토리 안에 SKILL.md와 references를 가지지만, Agents는 .md 파일 하나로 완성됩니다. 파일명이 곧 호출명이 됩니다. genie-space-designer.md → @genie-space-designer로 호출합니다.

    사전 준비: Unity Catalog MCP 서버

    @genie-space-designer가 테이블 스키마를 분석하려면 메타데이터 조회 도구가 필요합니다

    Tool설명예시

    describe_table 테이블 상세 메타데이터 조회 describe_table("main.default.t1") 
    
claude mcp add -s project unity-catalog -- \

    실습에 앞서 Unity Catalog MCP 서버를 설정합니다. 이 서버는 테이블 메타데이터를 조회하는 3개 도구를 제공합니다. list_schemas로 스키마 목록, list_tables로 테이블 목록, describe_table로 컬럼 상세 정보를 조회할 수 있습니다. 이 도구들이 있어야 genie-space-designer 에이전트가 테이블 스키마를 분석할 수 있습니다. exercise_unity_catalog_mcp.py는 완성 코드로 제공되므로 .mcp.json에 설정만 추가하면 됩니다.

    @agent는 한 번에 하나의 에이전트만 서브프로세스로 호출하는 방식입니다. 따라서 여러 에이전트를 체이닝하려면, @ 없이 자연어로 전체 워크플로우를 지시합니다. Claude Code는 .claude/agents/에 정의된 에이전트 역할을 참고하여 각 단계를 순차적으로 수행합니다. 이전 단계의 결과가 컨텍스트에 남아있으므로 자연스럽게 다음 단계에 전달됩니다. 이 방식이 실무에서 에이전트를 조합하는 가장 실용적인 패턴입니다.

    정리 & Q&A

    • Custom Agents: agent.md로 정의하는 역할 기반 AI 에이전트
    • 핵심 구조: 역할 + 제약 조건 + 출력 형식
    • Skills vs Agents: 절차 중심 vs 역할 중심, 실무에서는 조합 사용
    • 설계 원칙: 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임
    다음 섹션: Hooks — 이벤트 기반 자동화, settings.json 설정, 안전 가드레일

    질문이 있으신가요?

    정리하겠습니다. Custom Agents는 agent.md로 역할 기반 AI 에이전트를 정의합니다. 핵심 구조는 역할, 제약 조건, 출력 형식 세 가지입니다. Skills는 절차 중심, Agents는 역할 중심이며 실무에서는 조합하여 사용합니다. 설계 원칙은 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임입니다. 질문을 받겠습니다. 예상 질문들: - Q: Agent가 여러 개일 때 서로 호출할 수 있나요? A: 직접 호출은 불가능하지만, 사용자가 한 Agent의 결과를 다른 Agent에게 전달할 수 있습니다. designer → instructor처럼 순서대로 호출하면 됩니다. - Q: Skills와 Agents 중 어떤 것을 먼저 만들어야 하나요? A: 반복 작업이 먼저 보이면 Skills, 전문가 관점이 먼저 필요하면 Agents부터 시작하세요. - Q: tester 에이전트가 자동으로 Space 품질을 측정할 수 있나요? A: ask_genie로 질의하고 응답을 평가하는 방식으로 반자동 테스트가 가능합니다. 다음 섹션은 Hooks입니다. Claude Code의 세 번째 확장 체계로, 이벤트 기반 자동화를 다룹니다.
