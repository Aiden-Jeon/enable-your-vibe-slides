    Custom Agents란?

    • agent.md 파일로 정의하는 역할 기반 AI 에이전트
    • 프로젝트 내 .claude/agents/ 디렉토리에 위치
    • 특정 역할에 특화된 페르소나를 부여
    • 호출 방법: @agent-name
    핵심 아이디어: "이 작업은 이런 전문가 관점에서 해줘"를 코드화한 것 
    
# Claude Code에서 에이전트 호출
> @data-analyst fashion_recommendations 테이블의 브랜드별 추천 분포를 분석해줘
> @sql-reviewer 이 SQL 쿼리를 리뷰해줘
    Custom Agents의 핵심 개념을 설명합니다. agent.md는 마크다운 파일로, Claude Code에게 특정 역할의 전문가처럼 행동하도록 지시합니다. Skills가 작업 절차를 정의한다면, Agents는 역할과 관점을 정의합니다. 예를 들어 data-analyst 에이전트는 데이터 분석 전문가로, sql-reviewer 에이전트는 SQL 품질 검증 전문가로 동작합니다. 골뱅이(@) 기호로 호출하면 해당 역할에 맞는 응답을 받을 수 있습니다.

    agent.md 파일 구조

    
# 에이전트 이름

에이전트에 대한 한줄 설명

## 역할 (Role)
당신은 [전문 분야] 전문가입니다.
[역할에 대한 상세 설명]

## 제약 조건 (Constraints)
- 조사 범위 제한
- 사용 가능한 도구 명시
- 금지 사항

## 출력 형식 (Output Format)
결과물의 형태를 정의합니다:
- 마크다운 보고서
- 체크리스트

    역할 섹션

    에이전트의 전문성과 관점을 정의. "당신은 ~입니다"로 시작

    제약 조건 섹션

    에이전트가 하지 말아야 할 것, 범위 제한, 사용 도구 명시

    출력 형식 섹션

    결과물의 구조와 포맷을 미리 정의하여 일관성 보장

    agent.md의 기본 구조입니다. 역할 섹션에서는 에이전트의 전문성을 정의합니다. "당신은 ~ 전문가입니다"로 시작하는 것이 좋습니다. 제약 조건 섹션에서는 에이전트가 하지 말아야 할 것과 범위를 명시합니다. 출력 형식 섹션에서는 결과물이 어떤 형태여야 하는지 정의합니다. Skills의 SKILL.md와 비교하면, Skills는 처리 절차가 핵심이고, Agents는 역할과 제약 조건이 핵심입니다.

    실제 예시: data-analyst 에이전트

    Genie Space 데이터 분석 전문 에이전트

    
# data-analyst

Genie Space 데이터 분석 전문 에이전트

## 역할
당신은 Databricks SQL과 Genie Space를 활용하는
데이터 분석 전문가입니다.

## 제약 조건
- Genie MCP 서버의 ask_genie, continue_conversation만 사용
- 기존 Space에서 분석 (새 Space 생성 금지)
- 분석 결과에 사용된 SQL 반드시 포함
- 한국어로 결과물 작성

## 출력 형식
- **분석 주제**: [주제]
- **사용 SQL**: 생성된 쿼리
- **핵심 인사이트**: 불릿 포인트
- **추천 후속 분석**: 추가 분석 방향
    Genie Space 데이터 분석에 특화된 data-analyst 에이전트입니다. 역할에서 Databricks SQL과 Genie Space를 활용하는 데이터 분석 전문가로 정의했습니다. 제약 조건에서 Genie MCP 서버의 도구만 사용하고, 기존 Space에서만 분석하도록 제한했습니다. 출력 형식에서 분석 주제, 사용 SQL, 핵심 인사이트, 추천 후속 분석을 구조화했습니다. 이런 구조 덕분에 누가 @data-analyst를 호출해도 동일한 형식의 분석 결과를 받을 수 있습니다.

    실제 예시: sql-reviewer 에이전트

    SQL 쿼리 품질 검증 에이전트

    
# sql-reviewer

SQL 쿼리 품질 검증 에이전트

## 역할
당신은 Databricks SQL 성능 및 품질 검증
전문가입니다.

## 체크리스트
- [ ] SELECT *를 사용하지 않고 필요한 컬럼만 선택
- [ ] WHERE 절에 파티션 키 활용 여부
- [ ] JOIN 조건이 올바르고 불필요한 CROSS JOIN 없음
- [ ] GROUP BY와 집계 함수 매칭
- [ ] 데이터 타입 변환이 명시적
- [ ] LIMIT 절 포함 여부

## 출력 형식
- **정확성**: [쿼리 로직 검토]
- **성능**: [최적화 포인트]
- **개선 제안**: [구체적 수정 방안]
- **종합 평가**: PASS / NEEDS_IMPROVEMENT
    sql-reviewer 에이전트는 SQL 품질 검증에 특화되어 있습니다. 역할에서 Databricks SQL 성능 및 품질 검증 전문가로 정의했습니다. 체크리스트 섹션이 핵심인데, SQL 최적화 관점에서 검증해야 할 항목을 구체적으로 나열합니다. 출력 형식에서 정확성, 성능, 개선 제안, 종합 평가를 구조화했습니다. 이 에이전트를 사용하면 SQL 쿼리를 작성한 후 일관된 기준으로 품질을 검증할 수 있습니다.

    Skills vs Agents 비교

    SkillsAgents
    정의작업 절차 (How)역할과 관점 (Who)
    파일SKILL.mdagent.md
    위치.claude/skills/.claude/agents/
    호출/skill-name@agent-name
    초점"무엇을, 어떻게""누가, 어떤 범위에서"
    예시/genie-space, /data-summary@data-analyst, @sql-reviewer
    선택 가이드:
    반복적인 작업 절차가 있다 → Skill | 특정 전문가 관점이 필요하다 → Agent
    Skills와 Agents의 핵심 차이를 비교합니다. Skills는 작업의 절차, 즉 How를 정의합니다. "이 작업은 이 순서대로 해줘." Agents는 역할과 관점, 즉 Who를 정의합니다. "이 전문가처럼 행동해줘." 선택 기준은 간단합니다. 동일한 작업을 반복적으로 수행해야 한다면 Skill을 만듭니다. 특정 전문가의 관점이 필요하다면 Agent를 만듭니다. 실무에서는 둘을 조합해서 사용합니다. 예를 들어 @data-analyst에게 분석을 요청한 후, @sql-reviewer로 쿼리를 검증할 수 있습니다.

    Agent 설계 원칙

    1. 명확한 역할 정의

    "당신은 ~ 전문가입니다"로 시작. 역할이 모호하면 결과도 모호해집니다.

    2. 적절한 제약 조건

    하지 말아야 할 것을 명시. 범위가 넓을수록 제약 조건이 중요합니다.

    3. 구조화된 출력 형식

    결과물의 형태를 미리 정의하면 일관된 품질을 보장할 수 있습니다.

    4. 단일 책임 원칙

    하나의 Agent는 하나의 역할에 집중. 여러 역할이 필요하면 여러 Agent를 만듭니다.

    좋은 Agent를 설계하기 위한 네 가지 원칙입니다. 첫째, 역할을 명확하게 정의합니다. 모호한 역할은 모호한 결과를 낳습니다. 둘째, 적절한 제약 조건을 둡니다. 특히 범위가 넓은 에이전트일수록 제약이 중요합니다. 셋째, 출력 형식을 미리 정의합니다. 매번 다른 형식으로 결과가 나오면 활용하기 어렵습니다. 넷째, 단일 책임 원칙을 따릅니다. 리서치와 검증을 하나의 Agent에 넣지 마세요.

    ├   ├── data-analyst.md ← 데이터 분석 전문가
    ├   └── sql-reviewer.md ← SQL 검증 전문가
    ├── skills/ ← Skill 정의
    ├   ├── slide-deck/
    ├   ├── hands-on-code/
    ├   └── section-builder/
    └── CLAUDE.md ← 프로젝트 컨텍스트

    핵심 포인트

    • .claude/agents/ 디렉토리에 배치
    • 각 Agent는 하나의 .md 파일
    • 파일명 = 에이전트 호출명
    • Skills보다 구조가 단순

    Skills: 디렉토리 + SKILL.md + references/
    Agents: agent-name.md 파일 하나로 완성

    실제 프로젝트에서 Agent가 어떻게 배치되는지 보겠습니다. .claude 디렉토리 아래에 agents 폴더가 있고, 각 에이전트는 하나의 마크다운 파일입니다. Skills와 비교하면 구조가 훨씬 단순합니다. Skills는 디렉토리 안에 SKILL.md와 references를 가지지만, Agents는 .md 파일 하나로 완성됩니다. 파일명이 곧 호출명이 됩니다. data-analyst.md → @data-analyst로 호출합니다.

    • 출력 형식이 구조화되어 있는가?
    • 단일 책임 원칙을 따르는가?

    정리

    핵심 정리

    • Custom Agents — agent.md로 정의하는 역할 기반 AI 에이전트
    • 핵심 구조 — 역할 + 제약 조건 + 출력 형식
    • Skills vs Agents — 절차 중심 vs 역할 중심, 실무에서는 조합 사용
    • 설계 원칙 — 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임
    정리하겠습니다. Custom Agents는 agent.md 파일 하나로 역할 기반 AI 에이전트를 정의합니다. 핵심 구조는 역할, 제약 조건, 출력 형식 세 가지입니다. Skills는 절차 중심, Agents는 역할 중심으로, 실무에서는 둘을 조합해서 사용합니다. 좋은 Agent는 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임 원칙을 갖춥니다.
