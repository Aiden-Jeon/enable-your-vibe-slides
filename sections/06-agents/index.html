# Claude Code에서 에이전트 호출
> @genie-space-designer shared.fashion_recommendations의 테이블을 분석해서 Space를 설계해줘
> @genie-instructor 이 테이블 구성에 맞는 instructions를 작성해줘
    Subagents의 핵심 개념을 설명합니다. agent.md는 마크다운 파일로, Claude Code에게 특정 역할의 전문가처럼 행동하도록 지시합니다. Skills가 작업 절차를 정의한다면, Agents는 역할과 관점을 정의합니다. 예를 들어 genie-space-designer 에이전트는 Space 설계 전문가로, genie-instructor 에이전트는 instructions 작성 전문가로 동작합니다. 골뱅이(@) 기호로 호출하면 해당 역할에 맞는 응답을 받을 수 있습니다.

    Skills vs Agents 비교

    Agent를 배우기 전에, 이전 섹션의 Skills와 어떻게 다른지 먼저 짚어봅니다

    SkillsAgents
    정의작업 절차 (How)역할과 관점 (Who)
    파일SKILL.mdagent.md
    위치.claude/skills/.claude/agents/
    호출/skill-name@agent-name
    초점"무엇을, 어떻게""누가, 어떤 범위에서"
    구조디렉토리 + references/.md 파일 하나
    선택 가이드:
    반복적인 작업 절차가 있다 → Skill | 특정 전문가 관점이 필요하다 → Agent
    Agent를 본격적으로 배우기 전에, Skills와의 차이를 먼저 명확히 하겠습니다. Skills는 작업의 절차, 즉 How를 정의합니다. "이 작업은 이 순서대로 해줘." Agents는 역할과 관점, 즉 Who를 정의합니다. "이 전문가처럼 행동해줘." 구조적으로도 차이가 있습니다. Skills는 디렉토리에 SKILL.md와 references를 가지지만, Agents는 .md 파일 하나로 완성됩니다. 선택 기준은 간단합니다. 동일한 작업을 반복적으로 수행해야 한다면 Skill, 특정 전문가의 관점이 필요하다면 Agent를 만듭니다.

    같은 작업, 다른 접근

    Genie Space 만들기를 Skill과 Agent로 각각 어떻게 접근하는지 비교합니다

    Skill: /genie-space

    /genie-space "매출 분석" shared.sales

→ 자동으로:
  1. 스키마 분석
  2. instructions 작성
  3. Space 생성
  4. 테스트 질의
→ 완성된 Space 출력

    특징: 전 과정을 한 번에 자동 수행

    Agent: 3명의 전문가 순차 수행

    @designer 스키마 분석해줘
→ 설계 관점의 분석 리포트

@instructor instructions 작성해줘
→ 전문가 관점의 instructions

@tester 품질 테스트해줘
→ QA 관점의 평가 리포트

    특징: 각 단계를 전문가가 개별 수행

    실무 팁: 빠르게 반복할 때는 Skill, 품질을 세밀하게 통제할 때는 Agent, 실무에서는 둘을 조합하여 사용

    같은 Genie Space 만들기 작업을 Skill과 Agent로 각각 접근하는 방식을 비교합니다. Skill 방식은 /genie-space 명령 하나로 전 과정을 자동 수행합니다. 빠르고 일관적입니다. Agent 방식은 designer, instructor, tester 세 전문가가 각 단계를 개별 수행합니다. 각 단계에서 사람이 검토하고 피드백할 수 있습니다. 실무에서는 상황에 따라 선택합니다. 빠르게 반복할 때는 Skill, 품질을 세밀하게 통제할 때는 Agent를 사용합니다. 가장 좋은 방법은 둘을 조합하는 것입니다. 예를 들어 @researcher로 조사한 결과를 /genie-space Skill에 전달하는 식입니다.

    Agent 설계 원칙

    1. 명확한 역할 정의

    "당신은 ~ 전문가입니다"로 시작. 역할이 모호하면 결과도 모호해집니다.

    2. 적절한 제약 조건

    하지 말아야 할 것을 명시. 범위가 넓을수록 제약 조건이 중요합니다.

    3. 구조화된 출력 형식

    결과물의 형태를 미리 정의하면 일관된 품질을 보장할 수 있습니다.

    4. 단일 책임 원칙

    하나의 Agent는 하나의 역할에 집중. 여러 역할이 필요하면 여러 Agent를 만듭니다.

    좋은 Agent를 설계하기 위한 네 가지 원칙입니다. 첫째, 역할을 명확하게 정의합니다. 모호한 역할은 모호한 결과를 낳습니다. 둘째, 적절한 제약 조건을 둡니다. 특히 범위가 넓은 에이전트일수록 제약이 중요합니다. 셋째, 출력 형식을 미리 정의합니다. 매번 다른 형식으로 결과가 나오면 활용하기 어렵습니다. 넷째, 단일 책임 원칙을 따릅니다. 리서치와 검증을 하나의 Agent에 넣지 마세요.

    핵심 포인트

    • 각 Agent는 하나의 .md 파일
    • 파일명 = 에이전트 호출명
    • Skills보다 구조가 단순

    Skills: 디렉토리 + SKILL.md + references/
    Agents: agent-name.md 파일 하나로 완성

    실제 프로젝트에서 Agent가 어떻게 배치되는지 보겠습니다. .claude 디렉토리 아래에 agents 폴더가 있고, 각 에이전트는 하나의 마크다운 파일입니다. Skills와 비교하면 구조가 훨씬 단순합니다. Skills는 디렉토리 안에 SKILL.md와 references를 가지지만, Agents는 .md 파일 하나로 완성됩니다. 파일명이 곧 호출명이 됩니다. genie-space-designer.md → @genie-space-designer로 호출합니다.

# 에이전트 이름

에이전트에 대한 한줄 설명

## 역할 (Role)
당신은 [전문 분야] 전문가입니다.
[역할에 대한 상세 설명]

## 제약 조건 (Constraints)
- 조사 범위 제한
- 사용 가능한 도구 명시
- 금지 사항

## 출력 형식 (Output Format)
결과물의 형태를 정의합니다:
- 마크다운 보고서
- 체크리스트

    역할 섹션

    에이전트의 전문성과 관점을 정의. "당신은 ~입니다"로 시작

    제약 조건 섹션

    에이전트가 하지 말아야 할 것, 범위 제한, 사용 도구 명시

    출력 형식 섹션

    결과물의 구조와 포맷을 미리 정의하여 일관성 보장

    agent.md의 기본 구조입니다. 역할 섹션에서는 에이전트의 전문성을 정의합니다. "당신은 ~ 전문가입니다"로 시작하는 것이 좋습니다. 제약 조건 섹션에서는 에이전트가 하지 말아야 할 것과 범위를 명시합니다. 출력 형식 섹션에서는 결과물이 어떤 형태여야 하는지 정의합니다. Skills의 SKILL.md와 비교하면, Skills는 처리 절차가 핵심이고, Agents는 역할과 제약 조건이 핵심입니다.

    실제 예시: genie-space-designer 에이전트

    Genie Space 설계 전문 에이전트

    
# genie-space-designer
Genie Space 설계 전문 에이전트

## 역할
당신은 Databricks Genie Space 아키텍트입니다.
테이블 스키마를 분석하고 최적의 Space 구성을 설계합니다.

## 제약 조건
- 테이블 스키마 분석만 수행 (Space 생성 금지)
- 테이블은 5개 이하로 추천
- 한국어로 결과물 작성

## 출력 형식
- **분석 대상 테이블**: 스키마 요약
- **추천 테이블 구성**: 선정 이유 포함
- **Sample Questions**: 5개 이상
- **Space 방향성**: 분석 관점 제안
    Genie Space 설계에 특화된 genie-space-designer 에이전트입니다. 역할에서 Genie Space 아키텍트로 정의하여, 테이블 스키마를 분석하고 최적의 구성을 설계합니다. 제약 조건에서 스키마 분석만 수행하고, 실제 Space 생성은 하지 않도록 제한했습니다. 출력 형식에서 분석 대상 테이블, 추천 구성, Sample Questions, Space 방향성을 구조화했습니다. 이 에이전트는 Section 04에서 만든 Genie MCP의 첫 번째 활용 단계입니다.

# genie-instructor
Genie Space instructions 작성 전문 에이전트

## 역할
당신은 serialized_space instructions 블록 전문가입니다.
text_instructions, example_question_sqls,
sql_snippets를 작성합니다.

## 제약 조건
- instructions 블록 작성만 수행
- example_question_sqls 3개 이상 포함
- sql_snippets의 expressions, measures, filters 작성
- Space 생성이나 질의는 수행하지 않음

## 출력 형식
- **text_instructions**: 비즈니스 컨텍스트 설명
- **example_question_sqls**: 질문-SQL 쌍
- **sql_snippets**: expressions, measures, filters
- **join_specs**: 테이블 간 조인 조건
    genie-instructor 에이전트는 Genie Space instructions 작성에 특화되어 있습니다. 역할에서 serialized_space의 instructions 블록 전문가로 정의했습니다. 제약 조건에서 instructions 블록 작성만 수행하고, Space 생성이나 질의는 하지 않도록 제한했습니다. 출력 형식에서 text_instructions, example_question_sqls, sql_snippets, join_specs를 구조화했습니다. designer가 설계한 Space 구성을 기반으로, instructor가 실제 instructions을 작성하는 역할 분담입니다.

# genie-tester
Genie Space 품질 테스트 전문 에이전트

## 역할
당신은 Genie Space QA 전문가입니다.
ask_genie 도구로 질의하고 응답 품질을 평가합니다.

## 제약 조건
- ask_genie 도구로 테스트 질의만 수행
- Space 설정이나 instructions 수정 금지
- sample questions 기반으로 최소 5개 질의
- 한국어로 평가 리포트 작성

## 출력 형식
- **테스트 질의 목록**: 질문과 응답 쌍
- **정확성 평가**: 각 응답의 정확도 (O/X)
- **완전성 평가**: 누락된 정보 여부
- **개선 제안**: instructions 보완 포인트
    마지막 genie-tester 에이전트입니다. 역할에서 Genie Space QA 전문가로 정의하여, ask_genie 도구로 질의하고 응답 품질을 평가합니다. 제약 조건에서 테스트 질의만 수행하고, Space 설정이나 instructions 수정은 하지 않도록 제한했습니다. 출력 형식에서 테스트 질의 목록, 정확성 평가, 완전성 평가, 개선 제안을 구조화했습니다. 이 에이전트가 개선 제안을 내놓으면, 다시 instructor에게 전달하여 instructions를 보완하는 반복 루프가 가능합니다.

# 라이프사이클 순서대로 호출
> @genie-space-designer shared.fashion_recommendations의 테이블을 분석해서 Space를 설계해줘
> @genie-instructor designer 결과를 기반으로 instructions를 작성해줘
> @genie-tester Space에 sample questions를 질의하고 응답 품질을 평가해줘
    세 에이전트가 Genie Space 라이프사이클의 각 단계를 담당합니다. Phase 1: designer가 테이블 스키마를 분석하고 최적의 Space 구성을 설계합니다. Phase 2: instructor가 designer의 결과를 기반으로 text_instructions, example_sqls, sql_snippets를 작성합니다. Phase 3: tester가 완성된 Space에 ask_genie로 질의하고, 응답 품질을 평가합니다. genie-tester 에이전트는 ask_genie 도구로 sample questions를 질의하고, 응답의 정확성과 완전성을 평가하는 역할입니다. 이렇게 Section 04에서 만든 MCP 도구, Section 05에서 만든 Skill, Section 06에서 만든 Agents가 하나의 워크플로우로 연결됩니다.

    명시적 호출의 한계

    @agent는 한 번에 하나만 호출할 수 있습니다

    > @genie-space-designer 테이블 분석해줘     ← 1번째 입력, 결과 확인
> @genie-instructor instructions 작성해줘   ← 2번째 입력, 결과 확인
> @genie-tester 품질 테스트해줘              ← 3번째 입력, 결과 확인
    • @agent는 서브프로세스로 실행 — 한 번에 하나만 가능
    • 3단계 워크플로우 = 3번 수동 입력 필요
    • 단계마다 사람이 결과를 확인하고 다음을 호출해야 함

    세밀한 통제에는 좋지만, 전체 흐름을 한 번에 돌리고 싶다면?

    명시적 @호출은 세밀한 통제에 좋지만 한계가 있습니다. @agent는 서브프로세스로 실행되기 때문에 한 번에 하나만 호출할 수 있습니다. 3단계 워크플로우를 실행하려면 3번 수동으로 입력해야 합니다. 각 단계마다 사람이 결과를 확인하고 다음 에이전트를 호출해야 하죠. 그렇다면 전체 흐름을 한 번에 돌리는 방법은 없을까요?

    자연어로 자동 위임

    @ 없이 자연어만으로 전체 워크플로우를 지시할 수 있습니다

    > shared.fashion_recommendations 테이블을 분석해서 Space 설계하고,
  instructions 작성하고, 테스트까지 해줘.
  .claude/agents/ 에 있는 에이전트 정의를 참고해서 진행해줘.
    designer 자동 매칭
    →
    instructor 자동 매칭
    →
    tester 자동 매칭

    공식 문서: "Claude automatically delegates tasks based on the task description in your request, the description field in subagent configurations, and current context."
    — Claude Code Docs: Sub-agents

    해결책은 자연어 위임입니다. @agent-name 없이 자연어로 전체 작업을 설명하면, Claude가 알아서 적절한 Agent를 순차적으로 호출합니다. "테이블 분석해서 Space 설계하고, instructions 작성하고, 테스트까지 해줘"라고 하면, Claude가 designer → instructor → tester 순서로 자동 체이닝합니다. 이 자동 위임은 agent.md의 description 필드를 기반으로 요청을 매칭합니다. 따라서 description을 명확하게 작성하는 것이 자동 위임의 품질을 결정합니다.

    자동 위임의 동작 원리

    자연어 위임이 가능한 이유: 순차 실행과 컨텍스트 전달

    순차 실행

    Agent는 한 번에 하나만 실행됩니다. 여러 Agent가 필요하면 자동으로 순차 호출합니다.

    컨텍스트 전달

    같은 대화 안에서 이전 Agent의 결과가 컨텍스트에 유지되어 다음 Agent가 활용합니다.

    designer 결과 출력
    →
    컨텍스트
    전달
    →
    instructor 결과 참조
    →
    컨텍스트
    전달
    →
    tester 결과 참조

    새 대화를 시작하면 컨텍스트가 초기화됩니다. Agent 체이닝은 반드시 같은 대화에서 진행하세요.

    자연어 위임이 가능한 이유를 두 가지로 설명합니다. 첫째, Agent는 내부적으로 한 번에 하나만 실행됩니다. 병렬 실행이 아니라 순차 실행입니다. 둘째, 같은 대화 안에서 이전 Agent의 결과가 컨텍스트에 남아 다음 Agent에게 자연스럽게 전달됩니다. 이 두 가지 특성 덕분에 자연어 한 문장으로도 designer → instructor → tester 체이닝이 가능합니다. 단, 새 대화를 시작하면 컨텍스트가 초기화되므로, 체이닝 작업은 반드시 같은 대화에서 진행해야 합니다.

    정리 & Q&A

    • Agents: agent.md로 정의하는 역할 기반 AI 에이전트
    • 핵심 구조: 역할 + 제약 조건 + 출력 형식
    • Skills vs Agents: 절차 중심 vs 역할 중심 — 빠른 반복은 Skill, 세밀한 통제는 Agent, 실무에서는 조합
    • 설계 원칙: 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임

    질문이 있으신가요?

    ← 홈으로 돌아가기 다음 섹션 이동 →
    정리하겠습니다. Subagents는 agent.md로 역할 기반 AI 에이전트를 정의합니다. 핵심 구조는 역할, 제약 조건, 출력 형식 세 가지입니다. Skills는 절차 중심, Agents는 역할 중심이며 실무에서는 조합하여 사용합니다. 설계 원칙은 명확한 역할, 적절한 제약, 구조화된 출력, 단일 책임입니다. 질문을 받겠습니다. 예상 질문들: - Q: Agent가 여러 개일 때 서로 호출할 수 있나요?
