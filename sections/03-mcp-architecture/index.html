    MCP란?

    Model Context Protocol

    AI 모델이 외부 도구 및 데이터 소스와 소통하는 표준 프로토콜

    • Anthropic이 2024년 11월 공개한 오픈 프로토콜
    • AI 모델과 외부 세계 사이의 "USB-C" 같은 존재
    • 어떤 AI 모델이든, 어떤 도구든 하나의 규격으로 연결
    MCP는 Model Context Protocol의 약자입니다. Anthropic이 2024년 11월에 공개한 오픈 프로토콜로, AI 모델이 외부 도구나 데이터 소스와 표준화된 방식으로 소통할 수 있게 해줍니다. 쉽게 비유하면, 예전에 핸드폰마다 다른 충전 케이블을 쓰다가 USB-C 하나로 통일된 것처럼, AI와 도구 연결을 하나의 규격으로 통일하는 것입니다.

    MCP 핵심 구조

    N x M 문제 해결

    Claude GPT Gemini
    →
    MCP
    →
    Slack GitHub DB
    N + M 통합으로 해결

    Host / Client / Server

    Host (Claude Code)
    Client A Client B
    ↕ JSON-RPC 2.0
    Server A Server B
    MCP의 핵심 구조를 두 가지 관점에서 봅시다. 왼쪽은 N x M 문제 해결입니다. MCP가 없으면 모델마다 각 도구에 맞는 통합을 별도로 만들어야 하지만, MCP를 사용하면 N + M개의 통합만으로 모든 조합이 가능합니다. 오른쪽은 아키텍처입니다. Host(Claude Code) 안에 여러 Client가 있고, 각 Client는 하나의 Server와 1:1로 연결되어 JSON-RPC 2.0으로 통신합니다.

    MCP 서버의 두 가지 접근

    Managed MCP

    이미 만들어진 MCP 서버를
    설치만 하면 바로 사용

    • 3rd-party 공식 서버
    • Databricks 제공 서버

    Custom MCP

    비즈니스 요구에 맞게
    직접 개발하는 MCP 서버

    • 내부 API 연동
    • 도메인 특화 도구
    MCP 서버에는 크게 두 가지 접근 방식이 있습니다. Managed MCP는 이미 만들어져 있는 MCP 서버를 설치만 하면 바로 사용할 수 있습니다. 3rd-party 벤더가 제공하는 공식 서버나 Databricks가 제공하는 서버가 여기에 해당합니다. Custom MCP는 비즈니스 요구에 맞게 직접 개발하는 MCP 서버입니다. 내부 API 연동이나 도메인 특화 도구를 만들 때 필요합니다.

    Managed MCP: 3rd-party

    주요 서비스들이 공식 MCP 서버를 제공합니다:

    서비스 제공 기능 사용 예시
    GitHub PR, Issue, 코드 검색 "이 PR의 변경사항 요약해줘"
    Slack 메시지 읽기/쓰기 "#general 채널에 배포 알림 보내줘"
    Sentry 에러 모니터링 "최근 24시간 에러 분석해줘"
    Playwright 브라우저 자동화 "이 페이지 스크린샷 찍어줘"

    claude mcp add 명령으로 간단히 설치 → 바로 사용 가능

    먼저 3rd-party Managed MCP를 살펴봅시다. GitHub MCP 서버를 설치하면 PR이나 Issue를 자연어로 관리할 수 있고, Slack MCP를 설치하면 Claude Code에서 직접 메시지를 보낼 수 있습니다. Sentry로 에러를 분석하거나, Playwright로 브라우저를 자동화할 수도 있습니다. 이런 서버들은 claude mcp add 명령으로 간단히 설치하면 바로 사용할 수 있습니다. 직접 코드를 작성할 필요가 없다는 것이 Managed MCP의 큰 장점입니다.

    Managed MCP: Databricks

    Databricks가 네이티브로 제공하는 Managed MCP 서버 — UC 권한이 자동 적용됩니다

    서버기능접근
    UC FunctionsUC 함수를 MCP 도구로 실행실행
    Genie자연어 데이터 질의읽기
    SQLAI 생성 SQL 실행읽기/쓰기
    Vector Search벡터 인덱스 유사 문서 검색읽기
     
    // .mcp.json — Claude Code 연결 예시
{ "databricks-sql": { "type": "streamable-http",
    "url": "https://<workspace>/api/2.0/mcp/sql",
    "headers": { "Authorization": "Bearer ..." } } }

    공식 문서  |  외부 클라이언트 연결 가이드

    Databricks가 네이티브로 제공하는 Managed MCP 서버는 현재 4가지입니다. UC Functions는 Unity Catalog에 등록된 함수를 MCP 도구로 실행합니다. Genie는 Genie Space를 통해 자연어로 데이터를 질의합니다. 읽기 전용입니다. SQL은 AI가 생성한 SQL을 직접 실행할 수 있으며 Claude Code, Cursor 등과 연동됩니다. Vector Search는 벡터 인덱스를 검색해 유사한 문서를 찾습니다. Managed Embeddings 필수입니다. 모든 서버에 Unity Catalog 권한이 자동 적용되어, 사용자나 에이전트가 허가된 데이터만 접근합니다. Claude Code에서 연결하려면 .mcp.json에 streamable-http 타입으로 워크스페이스 URL을 지정하면 됩니다. 인증은 PAT 또는 OAuth를 사용할 수 있고, 프로덕션에서는 OAuth가 권장됩니다. 공식 문서 링크와 외부 클라이언트 연결 가이드를 참고하세요.

    Custom MCP: 왜 필요한가?

    Managed MCP만으로는 커버할 수 없는 영역이 있습니다:

    1
    내부 API 연동 — 사내 시스템, 레거시 API, 커스텀 서비스
    2
    비즈니스 로직 캡슐화 — 승인 워크플로우, 데이터 검증, 규칙 엔진
    3
    도메인 특화 도구 — ML 파이프라인 관리, 데이터 품질 체크, ETL 오케스트레이션

    Managed MCP = 범용 도구  /  Custom MCP = 우리 팀만의 도구

    그렇다면 Custom MCP는 왜 필요할까요? 첫째, 사내 시스템이나 레거시 API 등 내부 API를 연동해야 할 때입니다. 이런 시스템은 공식 MCP 서버가 존재하지 않습니다. 둘째, 승인 워크플로우나 데이터 검증 같은 비즈니스 로직을 캡슐화할 때입니다. 셋째, ML 파이프라인 관리나 데이터 품질 체크 같은 도메인 특화 도구가 필요할 때입니다. Managed MCP가 범용 도구라면, Custom MCP는 우리 팀만의 맞춤 도구입니다.

    Managed vs Custom 비교

    Managed MCP Custom MCP
    설치 claude mcp add 직접 개발 + 배포 후
    claude mcp add
    관리 벤더가 업데이트 자체 유지보수
    유연성 제공된 기능만 사용 무한한 커스터마이징
    보안 벤더의 보안 정책 자체 보안 제어
    적합한 경우 범용 SaaS 연동 내부 시스템, 도메인 로직
    Managed와 Custom을 비교해봅시다. 설치 면에서 둘 다 claude mcp add로 연결하지만, Custom은 서버를 직접 개발하고 배포하는 과정이 추가됩니다. 관리 면에서 Managed는 벤더가 업데이트해주지만, Custom은 자체 유지보수가 필요합니다. 유연성에서는 Custom이 압도적으로 유리합니다. 원하는 기능을 자유롭게 만들 수 있습니다. 보안에서도 Custom은 자체 보안 정책을 적용할 수 있어 민감한 데이터 처리에 적합합니다. 실전에서는 둘 중 하나만 쓰는 게 아니라, Managed와 Custom을 조합해서 사용합니다.

    Databricks가 제공하는 가치

    데이터 거버넌스

    Unity Catalog
    접근 제어 · 리니지

    AI 모델

    Foundation Models
    Model Serving

    인프라

    Databricks Apps
    서버리스 배포

    ↓
    MCP 서버를 만들고 → 거버넌스 아래에서 → 서버리스로 배포
    Databricks가 MCP 생태계에서 제공하는 가치는 세 가지가 결합된 것입니다. 첫째, Unity Catalog 기반의 데이터 거버넌스로 접근 제어와 리니지를 관리합니다. 둘째, Foundation Models와 Model Serving으로 AI 모델을 제공합니다. 셋째, Databricks Apps로 서버리스 인프라를 제공합니다. 이 세 가지가 결합되면, Custom MCP 서버를 만들어서 거버넌스 정책 아래에서 안전하게 서버리스로 배포할 수 있습니다. 다른 플랫폼에서는 이 세 가지를 각각 별도로 구축해야 합니다.

    이런 MCP를 만들 수 있습니다

    Databricks 위에서 Custom MCP로 만들면 효과적인 사례:

    파이프라인 모니터

    "어제 실패한 ETL 잡 알려줘"
    → Jobs API 연동

    비용 트래커

    "이번 달 팀 클러스터 비용 분석해줘"
    → system.billing 테이블 조회

    데이터 품질 체커

    "주문 테이블 품질 체크해줘"
    → 검증 규칙 실행

    모델 레지스트리

    "프로덕션 모델 최근 성능 보여줘"
    → MLflow API 연동

    모두 자연어 한 마디로 실행 — MCP가 복잡한 API 호출을 대신합니다

    Databricks 위에서 Custom MCP를 만들면 어떤 것들이 가능한지 구체적 사례를 봅시다. 파이프라인 모니터는 Jobs API를 연동해서 실패한 ETL 잡을 자연어로 조회합니다. 비용 트래커는 system.billing 시스템 테이블을 조회해서 클러스터 비용을 분석합니다. 데이터 품질 체커는 Great Expectations 같은 검증 규칙을 MCP 도구로 래핑합니다. 모델 레지스트리는 MLflow API를 연동해서 모델 성능 지표를 확인합니다. 이 모든 것들이 자연어 한 마디로 가능해집니다. 핵심은 복잡한 API 호출을 MCP가 추상화하고, 사용자는 자연어로만 요청하면 된다는 것입니다.

    FastMCP 소개

    FastMCP: Python으로 MCP 서버를 가장 쉽게 만드는 방법

    특징

    • 데코레이터 기반의 직관적 API
    • 타입 힌트에서 자동으로 스키마 생성
    • docstring이 도구 설명이 됨
    • uv로 의존성 없이 바로 실행

    설치

    # 프로젝트에 추가
uv add fastmcp

# 또는 프로젝트 전체 의존성 설치
uv sync
    FastMCP는 Python으로 MCP 서버를 가장 쉽게 만들 수 있는 라이브러리입니다. FastAPI에서 영감을 받아, 데코레이터 기반의 직관적인 API를 제공합니다. Python 타입 힌트에서 자동으로 JSON Schema를 생성하고, 함수의 docstring이 도구의 설명으로 자동 변환됩니다. uv를 사용하면 가상환경 설정 없이 바로 실행할 수 있어 매우 편리합니다.

    FastMCP 핵심 패턴

    from fastmcp import FastMCP

mcp = FastMCP("서버 이름")           # 1. 서버 인스턴스 생성

@mcp.tool()                           # 2. 도구 등록
def my_tool(param: str) -> str:       #    타입 힌트 → JSON Schema
    """도구 설명 (AI가 읽는 텍스트)"""  #    docstring → description
    return f"결과: {param}"

if __name__ == "__main__":
    mcp.run()                         # 3. 서버 실행

    3단계 패턴

    1. FastMCP("이름") — 서버 인스턴스 생성
    2. @mcp.tool() — 데코레이터로 도구 등록
    3. mcp.run() — 서버 실행
    FastMCP로 MCP 서버를 만드는 핵심 패턴은 3단계입니다. 첫째, FastMCP 인스턴스를 생성합니다. 둘째, @mcp.tool() 데코레이터로 함수를 도구로 등록합니다. 이때 타입 힌트가 자동으로 JSON Schema가 되고, docstring이 도구 설명이 됩니다. 셋째, mcp.run()으로 서버를 실행합니다. 이 패턴만 기억하면 어떤 MCP 서버든 만들 수 있습니다. 실습에서 이 패턴을 직접 적용해봅니다.

    실습: Claude Code로 MCP 서버 만들기!

    실습 순서 (10분)

    1. 스켈레톤 확인: exercise 파일의 TODO 주석 확인
    2. Claude Code로 구현: TODO를 구현해달라고 요청
    3. 테스트: MCP 서버 실행 및 Claude Code 연결

    Claude Code 워크플로우

    # 1. 프로젝트 루트에서 Claude Code 실행
claude

# 2. @로 파일 컨텍스트 추가하여 구현 요청
> @03-mcp-architecture/exercise_01_hello_mcp.py 의 TODO를 구현해줘

# 3. MCP 서버 테스트 실행
uv run --with fastmcp fastmcp run \
    03-mcp-architecture/exercise_01_hello_mcp.py
    이제 직접 실습해봅시다! 총 10분 정도 시간을 드립니다. 이번 실습은 Vibe Coding 방식으로 진행합니다. 중요한 점은 프로젝트 루트에서 Claude Code를 실행하는 것입니다. cd로 하위 디렉토리에 들어가면 프로젝트 루트의 CLAUDE.md나 .mcp.json 등 프로젝트 설정이 적용되지 않습니다. 대신 @를 사용해서 파일 컨텍스트를 직접 추가하면 프로젝트 설정이 유지되면서도 원하는 파일을 참조할 수 있습니다. 시간이 남으면 Exercise 02도 같은 방식으로 진행하세요.
