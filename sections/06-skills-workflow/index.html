    Skills로 워크플로우 구조화


    Enable Your Vibe — AI-Powered Development Hands-on

    25분 | 실습

    섹션 6을 시작합니다. 이번 세션에서는 Skills를 활용하여 개발 워크플로우를 구조화하는 방법을 배웁니다. 앞서 섹션 4에서 Skills 생태계의 개념을 배웠다면, 이번 세션에서는 직접 나만의 Skill을 설계하고 만들어보는 실습을 진행합니다. 참가자들에게 질문: "반복적으로 수행하는 개발 작업이 있으신가요? 예를 들어 API 엔드포인트 추가, 데이터 처리 파이프라인 작성 등."

    학습 목표

    1. Skill 설계 원칙을 이해하고 Claude Code를 활용하여 SKILL.md를 작성할 수 있다
    2. 레퍼런스 파일을 활용하여 일관된 품질의 코드를 생성할 수 있다
    3. Skill 체이닝 패턴으로 복잡한 워크플로우를 구성할 수 있다
    4. 직접 Custom Skill을 만들어 팀에서 활용할 수 있다
    이번 세션이 끝나면 참가자들은 네 가지 역량을 갖추게 됩니다. 첫째, Skill을 어떻게 설계하는지 원칙을 이해하고 Claude Code를 활용하여 SKILL.md 파일을 작성할 수 있습니다. 둘째, 레퍼런스 파일을 활용해서 AI가 항상 팀 컨벤션에 맞는 코드를 생성하게 할 수 있습니다. 셋째, 여러 Skill을 연결하는 체이닝 패턴을 이해합니다. 넷째, 실습을 통해 실제로 Custom Skill을 만들어봅니다.

    왜 Skills가 필요한가?

    반복 작업의 자동화

    • 매번 같은 프롬프트를 반복 작성
    • 동일한 패턴의 코드를 매번 설명
    • 컨벤션을 매번 상기시켜야 함
    • 실수와 누락이 발생

    팀 표준화

    • 팀원마다 다른 프롬프트 사용
    • 코드 스타일이 일관되지 않음
    • 베스트 프랙티스 공유가 어려움
    • 온보딩에 시간이 많이 소요

    Skills = 반복 작업을 표준화된 명령으로 캡슐화

    Skills가 왜 필요한지 두 가지 관점에서 살펴봅시다. 첫째, 반복 작업의 자동화입니다. 매번 같은 프롬프트를 작성하고, 같은 패턴의 코드를 설명하는 것은 비효율적입니다. 게다가 매번 컨벤션을 상기시켜야 하고, 그 과정에서 실수가 발생할 수 있습니다. 둘째, 팀 표준화입니다. 팀원마다 다른 프롬프트를 사용하면 코드 품질이 들쭉날쭉해집니다. Skills는 이런 문제를 해결합니다. 검증된 프롬프트와 레퍼런스를 하나의 명령으로 캡슐화하여, 누구나 동일한 품질의 결과물을 얻을 수 있게 합니다.

    Skill 설계 원칙

    1. 하나의 Skill = 하나의 명확한 목적

    Skill은 한 가지 작업만 수행해야 합니다. "API 엔드포인트 생성", "데이터 요약 보고서 작성" 등 목적이 명확해야 합니다.

    2. 입력 → 처리 → 출력 구조

    사용자 입력이 무엇이고, 어떤 처리를 거쳐, 어떤 결과물이 나오는지 명확히 정의합니다.

    3. 레퍼런스 파일로 품질 보장

    컨벤션, 템플릿, 예시 파일을 레퍼런스로 제공하면 AI가 일관된 품질의 코드를 생성합니다.

    좋은 Skill = 명확한 목적 + 구조화된 절차 + 레퍼런스 기반 품질

    Skill을 잘 설계하기 위한 세 가지 핵심 원칙입니다. 첫째, 하나의 Skill은 하나의 명확한 목적만 가져야 합니다. 너무 많은 기능을 하나에 넣으면 복잡해지고 유지보수가 어렵습니다. Unix 철학의 "한 가지를 잘 하라"와 같은 맥락입니다. 둘째, 입력-처리-출력 구조를 명확히 정의해야 합니다. 함수를 설계하듯이, Skill도 인터페이스가 명확해야 합니다. 셋째, 레퍼런스 파일을 적극 활용해야 합니다. AI에게 "이런 스타일로 만들어줘"라고 말하는 것보다, 실제 컨벤션 문서나 예시 코드를 레퍼런스로 제공하는 것이 훨씬 정확한 결과를 냅니다.

    SKILL.md 작성법

    SKILL.md는 Claude Code에게 Skill의 동작 방식을 알려주는 명세서입니다

    사용법 섹션

    Skill을 호출하는 방법과 전달할 인자를 정의

    ## 사용법
/api-endpoint [엔드포인트설명]

    처리 절차 섹션

    Skill이 수행할 단계별 작업을 순서대로 기술

    ## 처리 절차
1. 레퍼런스 파일 읽기
2. 입력 분석
3. 코드 생성

    출력 형식 섹션

    최종 결과물의 형태와 포맷을 명시

    ## 출력
- 엔드포인트 코드 (Python)
- Pydantic 모델 (Python)

    전체 구조 예시

    # skill-name 스킬
설명 한 줄

## 사용법
## 처리 절차
## 출력 형식
    SKILL.md의 각 섹션을 자세히 살펴보겠습니다. 사용법 섹션에서는 슬래시 명령과 인자를 정의합니다. 사용자가 어떻게 호출하는지 명확히 해야 합니다. 처리 절차 섹션은 가장 중요한 부분입니다. AI가 어떤 순서로 작업을 수행할지 단계별로 기술합니다. 여기서 레퍼런스 파일을 읽으라는 지시도 포함할 수 있습니다. 출력 형식 섹션에서는 최종 결과물이 어떤 형태인지 명시합니다. 코드 파일인지, 마크다운 문서인지, 어떤 구조를 따르는지 등을 정의합니다. 이 세 가지 섹션만 잘 작성하면 기본적인 Skill을 만들 수 있습니다.

    레퍼런스 파일 활용

    레퍼런스 파일은 AI에게 "이 기준에 맞춰 생성하라"고 알려주는 가이드입니다

    conventions (컨벤션)

    코딩 스타일, 네이밍 규칙, 에러 처리 방식 등 팀의 표준을 정의합니다

    templates (템플릿)

    코드 생성의 기반이 되는 골격 코드를 제공합니다

    examples (예시)

    완성된 코드 예시를 보여주어 AI가 패턴을 학습하게 합니다

    # api-conventions.md

## 응답 형식
{
  "status": "success" | "error",
  "data": { ... },
  "message": "설명"
}

## 네이밍 규칙
- 엔드포인트: kebab-case
- 함수명: snake_case
- 모델명: PascalCase

## 에러 처리
- HTTPException 사용
- 400, 404, 500 상태 코드
    레퍼런스 파일은 Skill의 품질을 결정짓는 핵심 요소입니다. 세 가지 유형의 레퍼런스 파일이 있습니다. Conventions는 팀의 코딩 컨벤션을 정의합니다. 네이밍 규칙, 에러 처리 방식, 응답 형식 등을 담습니다. Templates는 코드의 골격을 제공합니다. AI가 이 골격을 기반으로 코드를 생성하므로 구조가 일관됩니다. Examples는 완성된 코드 예시입니다. AI는 예시를 참고하여 유사한 패턴의 코드를 생성합니다. 오른쪽에 보이는 것이 실제 API 컨벤션 레퍼런스 파일의 예시입니다. 이 파일을 Skill의 references 디렉토리에 넣어두면, AI가 항상 이 규칙에 맞춰 코드를 생성합니다.

    실제 Skill 구조 예시

    .claude/skills/
    ├── api-endpoint/
    ├   ├── SKILL.md ← 스킬 정의
    ├   └── references/
    ├       ├── api-conventions.md
    ├       ├── endpoint-template.py
    ├       └── example-endpoint.py
    ├── data-summary/
    ├   └── SKILL.md ← 간단한 스킬
    └── test-generator/
        ├── SKILL.md
        └── references/
            └── test-conventions.md

    핵심 포인트

    • .claude/skills/ 디렉토리에 배치
    • 각 Skill은 독립된 디렉토리
    • SKILL.md는 필수 파일
    • references/는 선택이지만 권장

    간단한 Skill: SKILL.md 하나만으로도 동작
    고품질 Skill: 레퍼런스 파일로 일관성 보장

    실제 프로젝트에서 Skill이 어떤 구조로 배치되는지 보겠습니다. .claude/skills 디렉토리 아래에 각 Skill별 디렉토리가 위치합니다. api-endpoint 스킬처럼 레퍼런스가 있는 스킬은 references 하위 디렉토리에 컨벤션, 템플릿, 예시 파일을 둡니다. data-summary 스킬처럼 간단한 경우에는 SKILL.md 하나만으로도 충분합니다. 핵심은 각 Skill이 독립된 디렉토리를 가진다는 것입니다. 이 구조를 지키면 Skill을 쉽게 추가, 수정, 공유할 수 있습니다.

    Skill 체이닝 패턴

    Skill A의 출력을 Skill B의 입력으로 연결하여 복잡한 워크플로우를 구성합니다

    Skill A data-summary
    →
    출력 → 입력 요약 보고서
    →
    Skill B api-endpoint
    →
    Skill C test-generator

    체이닝 예시: 기능 개발 워크플로우

    1. /api-endpoint "사용자 프로필 조회 API"
   → 엔드포인트 코드 생성

2. /test-generator "생성된 엔드포인트에 대한 테스트"
   → 테스트 코드 자동 생성

3. /api-docs "프로필 조회 API 문서화"
   → API 문서 생성

    체이닝의 핵심: 각 Skill의 출력 형식이 다음 Skill의 입력 형식과 호환되어야 합니다

    Skill 체이닝은 여러 Skill을 연결하여 복잡한 워크플로우를 구성하는 패턴입니다. 예를 들어, 기능 개발 워크플로우를 생각해봅시다. 먼저 api-endpoint 스킬로 엔드포인트 코드를 생성하고, 그 결과물을 기반으로 test-generator 스킬로 테스트 코드를 만들고, 마지막으로 api-docs 스킬로 API 문서를 생성합니다. 체이닝이 잘 동작하려면 각 Skill의 출력 형식이 다음 Skill의 입력으로 자연스럽게 연결되어야 합니다. 이것은 Skill을 설계할 때부터 염두에 두어야 하는 중요한 포인트입니다.

    실습: 나만의 Skill 만들기 Hands-on

    Exercise 1: 간단한 Skill (10분)

    1. exercise_01_simple_skill/SKILL.md 템플릿을 확인합니다 (TODO 항목 확인)

    > exercise_01_simple_skill/SKILL.md에 data-summary 스킬을 작성해줘

    2. Claude Code에게 레퍼런스를 반영한 SKILL.md 작성을 요청합니다
    3. /api-endpoint를 실행하여 생성된 코드를 테스트합니다

    > exercise_02_skill_with_refs/SKILL.md에 api-endpoint 스킬을 작성해줘

    체크리스트

    • SKILL.md에 사용법이 명확히 정의되어 있는가?
    • 처리 절차가 단계별로 기술되어 있는가?
    • 출력 형식이 명시되어 있는가?
    • 레퍼런스 파일이 실제로 참조되고 있는가?
    • Skill을 실행했을 때 일관된 결과가 나오는가?

    도전 과제: 자신의 업무에 맞는 Skill을 하나 새로 설계해보세요!

    이제 실습 시간입니다. 두 가지 Exercise를 진행합니다. 워크플로우는 동일합니다: 먼저 SKILL.md 템플릿의 TODO 항목을 확인하고, Claude Code에게 작성을 요청한 뒤, 실제로 실행하여 테스트합니다. Exercise 1은 간단한 Skill 만들기입니다. exercise_01_simple_skill 디렉토리의 SKILL.md 템플릿을 확인하고, Claude Code에게 "exercise_01_simple_skill/SKILL.md에 data-summary 스킬을 작성해줘"라고 요청합니다. Claude Code가 작성한 SKILL.md를 확인한 뒤, /data-summary 명령으로 직접 실행해봅니다. 약 10분 정도 소요됩니다. Exercise 2는 레퍼런스 파일이 있는 Skill입니다. exercise_02_skill_with_refs 디렉토리의 템플릿과 references 디렉토리를 먼저 확인하고, Claude Code에게 "exercise_02_skill_with_refs/SKILL.md에 api-endpoint 스킬을 작성해줘"라고 요청합니다. 레퍼런스 파일이 SKILL.md에 어떻게 반영되었는지 확인하고, /api-endpoint 명령으로 실행하여 테스트합니다. 약 15분 정도 소요됩니다. 체크리스트를 활용하여 Claude Code가 작성한 Skill이 올바르게 구성되었는지 검토해보세요.
